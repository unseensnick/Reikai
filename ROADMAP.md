Day-to-day session state lives in `Handoff.md`; durable architecture, decisions, and lessons live in the `.claude` memories; historical per-phase plans are archived (see the end). Supersedes the old `rosy-gathering-walrus.md` master plan.

## Active sequence (do in this order)

Two ordering rules: **the backup proto goes last** (it must serialize everything the earlier items add: history rows, tracker links, categories, merges, grouping), and the quick polish leads for momentum. The Tier 0 cleanup above is done, so these slices build on the shared helpers rather than extending copy-paste.

### 1. Novel grouping: collapse-at-bottom  ✅ done (`e5c0ecee6`)
Threaded the novel session-collapse set + the `collapsedDynamicAtBottom` pref through `buildState` into `buildNovelDynamicGrouping` (a 3-way inner combine on the main flow, so a collapse toggle rebuilds and re-sinks live); dropped the redundant post-build collapse stamp/collector. Also fixed `toggleAllCategoriesCollapsed` to key dynamic groups by `headerKey` (real categories by id), mirroring `LibraryScreenModel`, so the hopper "collapse all" collapses + sinks dynamic groups too. Verified on-device (Z Fold) in both the single-list and tabbed views.

### 2. Novel library: include/exclude category filter  ✅ done (`c0eecd59b`)
Added the category include/exclude row to the novel filter sheet and applied the shared `matchesCategoryFilter` in `NovelLibraryScreenModel` (its own novel-specific prefs + a dedicated picker-categories flow that includes the synthesized Default row; `State.displayedCategories` was the wrong source, it drops empties and flips to dynamic groups). Also extracted the category-filter row + picker dialog into a shared `CategoryFilterRow` (`reikai.presentation.category`) and re-pointed the manga library + Updates feed onto it (three copy-pasted impls collapsed to one, sync-neutral). On-device verified in both the tabbed + single-list views, plus manga / Updates regression.

### 3. Novel details: bulk-download dropdown  ✅ done (`5338240a3`)
Added the manga-style download dropdown (next 1/5/10/25, unread, bookmarked, the actual `DownloadAction` set, no "all") to `NovelToolbar`, wired to `NovelDetailsScreenModel.runDownloadAction`, which queues over the **full stored chapter list** (`getByNovelId`, all pages) via `NovelDownloadManager`, not just the on-screen page. Extracted the per-action selection into a shared, unit-tested `selectChaptersForDownloadAction` (`reikai.presentation.novel`) and re-pointed the library's `performDownloadAction` onto it. On-device verified (dropdown anchors under the toolbar icon) + library multi-select download regression.

### 4. Fix the dead "Last read" novel sort  ✅ done (`b183ba757`)
The novel library `LastRead` sort read `novels.last_read_at` but nothing wrote it, so it was a silent no-op. The reader now stamps `last_read_at` on every progress save via a focused single-column `setLastReadAt` query (no full-row rewrite), keyed to the chapter's owning novel (so a merged session stamps the source actually read), reader-only (mark-as-read checkboxes don't stamp), matching how Mihon's manga history keys on the chapter read. Also harmonized the merge `LastRead`: a merged entry now sorts by the most recent read across all members (group max in `NovelMergeCollapse` + `MangaMergeCollapse`), not just the display representative's own value, which fixed the same latent gap on the manga side. No migration (the column has existed since 14.sqm, the migration that creates the novels table). Unit-tested (the rollup on both collapse classes) and on-device verified (Z Fold): novel stamp + the `LastRead` sort flip, the merged-novel rollup (a broken-source NULL member + a read member), and merged-manga Apotheosis sorting to the top via a non-representative member's read. The read-time write-path is the hook #5 (History tab) builds on.

### 5. Novel History tab  ✅ done (`bc0b78d37`..`2d2807015`)
The History half of the old "unified Recents", folded into the existing History tab the way the Updates tab was consolidated: Mihon's untouched manga `HistoryScreenModel` and a new `NovelHistoryScreenModel` render through one `ReikaiHistoryScreen` behind the All / Manga / Novels chip (one search box + one clear-all drive both; rows interleave by read time, newest first, with shared date headers). The novel reader records history on chapter switch + on exit (a session timer accumulating `time_read`), the novel twin of `ReaderViewModel.updateHistory`. Data: a `novel_history` table + `novelHistoryView` (one row per novel, the max-last-read chapter) + migration `18.sqm`; `last_read` is plain epoch-millis (no Date adapter, matching `novels.last_read_at`). Resume reopens the recorded chapter if unread else the next (`GetNextNovelChapter`); tab-reselect resumes the globally-latest read across both libraries. Soft-delete one / reset-by-novel / hard clear-all, all chip-scoped. Built across six commits (data → domain → reader → UI → host → tests). Unit-tested (`GetNextNovelChapter`, the `buildHistoryRows` interleave) and on-device verified (Z Fold): migration on an existing install, switch+exit writes with duration, one-row-per-novel view, interleaved feed + date headers, chip filter, resume, delete-one fallback, and Novels-scoped clear-all leaving manga history intact.
- **One deliberate parity trim:** the novel history row has no add-to-library button (manga's does, for non-library entries); re-add via the cover → details screen. The manga add-favorite path drags in duplicate-check + category + tracking dialogs that are manga-specific.

### 6. Category bulk-delete  ✅ done (`5173d08b4`)
Long-press multi-select on the shared category manager with Mihon's action-mode toolbar (Select all / Invert / Delete), for both the Manga and Novels tabs. Delete is deferred with an Undo snackbar rather than an immediate DB write: rows are filtered out of the live category flow via a `pendingDeleteIds` set and only committed when the snackbar dismisses, so undo never needs a lossy re-insert (the category junction tables cascade their membership rows, and a re-insert gets a new id). The single-row delete was re-routed through the same deferred path so it is undoable too. Leaving the surface (tab switch or back) commits any pending delete in a `withNonCancellableContext` block, fixing the case where the cancelled snackbar coroutine silently dropped it. Selection set math is a shared, unit-tested `reikai.presentation.category.CategorySelection`. On-device verified (Z Fold): single + bulk delete, undo, select-all / invert, tab-switch and back-out commit, both tabs.

### 7. Novel source migration  ✅ done (`9756d2a76`)
A favorite-gated **Migrate** action in the novel details overflow, the novel twin of Mihon's manga migration (and modelled on LNReader's `migrateNovel`). It searches the title across novel sources (reusing the global-search fan-out behind a shared results composable, hiding the current source), and on picking a target materialises it (fetch + chapter sync) then shows a Mihon-style "Select data to include" dialog with a Copy vs Migrate choice. `MigrateNovelUseCase` copies per-chapter read / bookmark / scroll-progress matched by chapter number, moves categories, favorites the target, and on Migrate unfavorites the old novel + splits it out of any merge group. New-target-row + unfavorite-old (not in-place, not delete): safer, recoverable, matches Mihon. History-tab rows and tracks are intentionally not carried (parity with Mihon; tracks land with #8). Chapter-number match unit-tested; on-device verified (Z Fold): entry → search → target dialog → Copy/Migrate, read/bookmark/progress + categories carry. **Download re-queue follow-up done** (`fc6dabb03`): chapters that were offline on the old source are re-queued for download on the target (matched by chapter number, files re-fetched not copied, like LNReader); the manager skips ones the target already has. Pure `chaptersToRedownload` unit-tested.

### 8. Novel trackers  `[M]`  — AniList / MyAnimeList / MangaUpdates / Kitsu
Mirror LNReader's tracker set, reusing Mihon's existing tracker infrastructure (OAuth/token flows). No new tracker service.
- **Scope:** a novel tracking sheet on novel details that binds a novel to those four trackers and writes status / progress / score; novel-track persistence (a `novel_track` table + repo).
- **Done when:** a novel can be bound + updated on each of the four trackers and it round-trips; on-device verified.
- **Note:** dedicated LN trackers (NovelUpdates, MiraiList, Novel Trackr, RanobeDB, Hardcover) were researched (June 2026) and are NOT viable as on-device sync targets; see the `novel-tracking` memory. Hardcover is the only future watch item.

Fence Mihon backup-file edits with `// RK`. Note `BackupManga`/`BackupChapter` already carry `memo` at proto 112/13 (from the TachiyomiX sync) — novel additions use new classes/numbers.

## Signed release pipeline (Mihon AGP signing)  ✅ built 2026-06-17 (pending repo PAT + first-run verify)

Replaced the rebase stopgap (debug-signed release via the `// RK` island) with AGP-native signing plus two GitHub Actions workflows, and re-pointed the in-app updater at Reikai's own release repos. Shipped in commits `88ce9e674`, `dc695cefb`, `21bf270de`, `049645a41`:

- **Signing** (`app/build.gradle.kts`, `// RK`): AGP-native, adapted from Mihon `6552ffe31`. Reconfigures the `debug` signing config the release buildType references: in CI under `unseensnick/*` it decodes the keystore from the repo secrets, locally it reads `keystore.properties` (gitignored); with neither, dev builds stay debug-signed.
- **In-app updater** (`AppUpdateChecker.kt`, `// RK`): `GITHUB_REPO` → `unseensnick/Reikai` (release) / `unseensnick/Reikai-preview` (preview); both channels build with `-Penable-updater`.
- **`release.yml`** (stable): a `v*` tag or `workflow_dispatch` (version/message) builds signed `assembleRelease`, publishes a **draft** release to `unseensnick/Reikai` (review gate; the updater sees it once published), `reikai-*.apk` + sha256, notes from CHANGELOG.
- **`preview.yml`** (preview/nightly): build-on-push to `design/mihon-rebase` (+ manual) builds signed `assemblePreview`, publishes a non-prerelease to `unseensnick/Reikai-preview` via the `PREVIEW_REPO_TOKEN` PAT, tag `r{commitCount}`, commit-log notes, prunes to 28. `paths-ignore **.md` + `cancel-in-progress` keep it quiet.
- **`build_check.yml`** migrated off the `null2264` composite to official `setup-java` (Java 21) + `setup-gradle`. Java 21 verified to build Reikai clean.
- Decisions baked in: telemetry and foss skipped; the existing `preview` buildType is reused as the nightly channel (no new buildType); build-on-push requires the cross-repo PAT (the build runs in `Reikai`, so the signing secrets stay only here and `Reikai-preview` is a pure release bucket).

**Remaining (user):** create a fine-grained PAT with **Contents: write** on `unseensnick/Reikai-preview`, stored as the `PREVIEW_REPO_TOKEN` secret on `unseensnick/Reikai`. The `unseensnick/Reikai-preview` repo is already created.

**Verify (first run):** a push to `design/mihon-rebase` publishes a preview to `Reikai-preview`; a tag/`workflow_dispatch` release draft-publishes to `Reikai`; on-device, About → Check for updates finds the published release and prompts, on both a release and a preview install.

## Parked / not building

- **Dedicated LN trackers** (NovelUpdates / MiraiList / Novel Trackr / RanobeDB / Hardcover): not viable as of June 2026 (no sanctioned read+write API for on-device use). Re-check Hardcover only if it leaves beta and ships OAuth + allowlisting. Detail in the `novel-tracking` memory.
- **Y13** force side-nav rail, **Y17** DOKI theme, **Y18** in-app app-icon changer (P8, dropped; revivable — Y18 is blocked on Reikai-branded icon assets).
- **Y4** drag-sort, **Y5** staggered grid, **Y8** (= R16, duplicate), **Y19** stats drill-down (out of scope).
- **EPUB export** (LNReader has it; out of plan).
- **Novel recommendations / related carousel:** gated on novel trackers (getting the mainstream trackers to track LNs at all is unreliable) and LN sources don't expose related-title metadata, so it is not worth building unless novel tracking proves out.
- **Upcoming / release calendar for novels:** LN sources rarely expose a reliable release cadence, so an upcoming feed would be mostly empty; the calendar stays manga-only.
- **Saved searches** (browse filter presets): Tachiyomi/J2K-lineage power-user feature (name and store a per-source query+filter combo, re-applied in one tap). Not Reikai-distinctive, and was View/XML on the old browse screen, so closing it is a full Compose rebuild for low value. The DB + serializer layer (`saved_search` table + `FilterSerializer`) survives on `design/library-compose` if ever revived.
- **Dynamic launcher shortcuts** (`MangaShortcutManager`): J2K-lineage convenience that filled the long-press-app-icon menu with recent manga/sources. Built on the old Recents presenter + Conductor controllers + `PreferencesHelper`, so it's a rewire against Mihon's data layer, not a port, for a cosmetic nicety. Mihon ships only a static `shortcuts.xml`.
