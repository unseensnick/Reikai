# Reikai Roadmap

Forward-looking plan for the Mihon rebase (branch `design/mihon-rebase`, which becomes `main` when the rebase ships). This is the single source for what's left and in what order. Day-to-day session state lives in `Handoff.md`; durable architecture, decisions, and lessons live in the `.claude` memories; historical per-phase plans are archived (see the end). Supersedes the old `rosy-gathering-walrus.md` master plan.

### 4. Fix the dead "Last read" novel sort  ✅ done (`b183ba757`)
The novel library `LastRead` sort read `novels.last_read_at` but nothing wrote it, so it was a silent no-op. The reader now stamps `last_read_at` on every progress save via a focused single-column `setLastReadAt` query (no full-row rewrite), keyed to the chapter's owning novel (so a merged session stamps the source actually read), reader-only (mark-as-read checkboxes don't stamp), matching how Mihon's manga history keys on the chapter read. Also harmonized the merge `LastRead`: a merged entry now sorts by the most recent read across all members (group max in `NovelMergeCollapse` + `MangaMergeCollapse`), not just the display representative's own value, which fixed the same latent gap on the manga side. No migration (the column has existed since 14.sqm, the migration that creates the novels table). Unit-tested (the rollup on both collapse classes) and on-device verified (Z Fold): novel stamp + the `LastRead` sort flip, the merged-novel rollup (a broken-source NULL member + a read member), and merged-manga Apotheosis sorting to the top via a non-representative member's read. The read-time write-path is the hook #5 (History tab) builds on.

### 5. Novel History tab  ✅ done (`bc0b78d37`..`2d2807015`)
The History half of the old "unified Recents", folded into the existing History tab the way the Updates tab was consolidated: Mihon's untouched manga `HistoryScreenModel` and a new `NovelHistoryScreenModel` render through one `ReikaiHistoryScreen` behind the All / Manga / Novels chip (one search box + one clear-all drive both; rows interleave by read time, newest first, with shared date headers). The novel reader records history on chapter switch + on exit (a session timer accumulating `time_read`), the novel twin of `ReaderViewModel.updateHistory`. Data: a `novel_history` table + `novelHistoryView` (one row per novel, the max-last-read chapter) + migration `18.sqm`; `last_read` is plain epoch-millis (no Date adapter, matching `novels.last_read_at`). Resume reopens the recorded chapter if unread else the next (`GetNextNovelChapter`); tab-reselect resumes the globally-latest read across both libraries. Soft-delete one / reset-by-novel / hard clear-all, all chip-scoped. Built across six commits (data → domain → reader → UI → host → tests). Unit-tested (`GetNextNovelChapter`, the `buildHistoryRows` interleave) and on-device verified (Z Fold): migration on an existing install, switch+exit writes with duration, one-row-per-novel view, interleaved feed + date headers, chip filter, resume, delete-one fallback, and Novels-scoped clear-all leaving manga history intact.
- **One deliberate parity trim:** the novel history row has no add-to-library button (manga's does, for non-library entries); re-add via the cover → details screen. The manga add-favorite path drags in duplicate-check + category + tracking dialogs that are manga-specific.

### 6. Category bulk-delete  ✅ done (`5173d08b4`)
Long-press multi-select on the shared category manager with Mihon's action-mode toolbar (Select all / Invert / Delete), for both the Manga and Novels tabs. Delete is deferred with an Undo snackbar rather than an immediate DB write: rows are filtered out of the live category flow via a `pendingDeleteIds` set and only committed when the snackbar dismisses, so undo never needs a lossy re-insert (the category junction tables cascade their membership rows, and a re-insert gets a new id). The single-row delete was re-routed through the same deferred path so it is undoable too. Leaving the surface (tab switch or back) commits any pending delete in a `withNonCancellableContext` block, fixing the case where the cancelled snackbar coroutine silently dropped it. Selection set math is a shared, unit-tested `reikai.presentation.category.CategorySelection`. On-device verified (Z Fold): single + bulk delete, undo, select-all / invert, tab-switch and back-out commit, both tabs.

### 7. Novel source migration  ✅ done (`9756d2a76`)
A favorite-gated **Migrate** action in the novel details overflow, the novel twin of Mihon's manga migration (and modelled on LNReader's `migrateNovel`). It searches the title across novel sources (reusing the global-search fan-out behind a shared results composable, hiding the current source), and on picking a target materialises it (fetch + chapter sync) then shows a Mihon-style "Select data to include" dialog with a Copy vs Migrate choice. `MigrateNovelUseCase` copies per-chapter read / bookmark / scroll-progress matched by chapter number, moves categories, favorites the target, and on Migrate unfavorites the old novel + splits it out of any merge group. New-target-row + unfavorite-old (not in-place, not delete): safer, recoverable, matches Mihon. History-tab rows and tracks are intentionally not carried (parity with Mihon; tracks land with #8). Chapter-number match unit-tested; on-device verified (Z Fold): entry → search → target dialog → Copy/Migrate, read/bookmark/progress + categories carry. **Download re-queue follow-up done** (`fc6dabb03`): chapters that were offline on the old source are re-queued for download on the target (matched by chapter number, files re-fetched not copied, like LNReader); the manager skips ones the target already has. Pure `chaptersToRedownload` unit-tested.

### 8. Novel trackers  ✅ done (`5e8f53aca`, on-device verified)  — AniList / MyAnimeList / MangaUpdates / Kitsu
A Tracking overflow action on novel details opens a sheet (the novel twin of Mihon's manga tracking sheet, reusing the same `Tracker` services + OAuth) that binds a novel and writes status / progress / score / dates, persisted to the already-shipped `novel_tracks` table (queries + `NovelTrack` model + repo + interactors, mirroring `novel_history`). The four trackers' `search()` got a `// RK`-fenced novel path (`searchNovel`) since Mihon's manga search filters light novels out; everything downstream (find / update / OAuth) reuses Mihon as-is. Bind (`AddNovelTrack`) and the per-field updates (`NovelTrackUpdater`) are faithful ports of `AddTracks.bind` / `BaseTracker` that persist to `novel_tracks` instead of `manga_sync`. Auto-sync pushes progress from both the reader (≥97%, `autoUpdateTrack`) and details mark-read (`AutoTrackState` never / always / ask), with a novel offline-retry queue (`TrackNovelChapter` + `NovelDelayedTracking{Store,UpdateJob}`). The tracking UI reuses Mihon's composables verbatim via a `NovelTrack -> domain Track` carrier adapter (one `// RK` `allowPrivate` flag hides the private toggle novels lack). Unit-tested (conversion round-trip, updater transitions). Merged novels: tracking is **group-aware** (a track bound on one source advances/shows when reading or viewing any sibling source), centralized in `NovelMergeManager.relatedNovelIdsFor` + `GetNovelTracks.awaitGroup`/`subscribeGroup` (cleaner than manga's copy-to-each-member-at-bind: one row while merged, no duplicates/gates, possible because we own `TrackNovelChapter`). Unmerge-survival is handled by `PropagateNovelTrackerLinks` (the novel twin of manga's, but materialised lazily): at the split moment (library unmerge + details Manage-sources split, gated by `syncTrackerLinksGrouped`) it copies the group's trackers onto each favorited member so every source keeps the tracker after a split. **Deliberate trims:** no `private` listing (the table shipped without the column; no migration), no on-bind start-date backfill (still settable in the sheet).
- **Note:** dedicated LN trackers (NovelUpdates, MiraiList, Novel Trackr, RanobeDB, Hardcover) were researched (June 2026) and are NOT viable as on-device sync targets; see the `novel-tracking` memory. Hardcover is the only future watch item.

### 9. Novel backup proto (S9)  ✅ done (`a6027edf2`, on-device verified)  — last core item
The novel twin of Mihon's backup/restore, plus an installed-sources backup. New `BackupNovel` / `BackupNovelChapter` / `BackupNovelTracking` / `BackupNovelHistory` / `BackupNovelCategory` proto classes (700 range, clear of Mihon's 1-106 and Komikku's 600/610), grafted onto the root `Backup` and into `BackupCreator` / `BackupRestorer` with `// RK`. `NovelBackupCreator` / `NovelRestorer` mirror the manga path (match-or-insert by url+source, re-link chapters by url / categories by name / tracks by tracker id / history by chapter url), gated by the same `BackupOptions` toggles as manga (novels are first-class library content, no separate toggle). Merge/grouping survives id reassignment: each manual merge + unmerge group is serialized as stable `{url, source}` refs (`BackupNovelMerge`) and rebuilt against the restored ids, with the two id-keyed merge prefs excluded from the generic preference restore (`// RK` in `PreferenceRestorer`). One new `novel_history` query pair (read for backup, idempotent `restoreUpsert`). **Installed-sources follow-on (folded in):** `BackupExtension` records installed manga extensions and a restore reinstalls them via the standard installer (unmatched logged); novel plugins need no new proto (their state rides the preference backup) and are re-downloaded from their saved URLs by `NovelPluginRestorer`. Unit-tested (`NovelBackupRoundTripTest`: merge id-remap round-trip + proto mapping); on-device verified (Z Fold): a real backup includes the novel library (titles absent from the pre-novel backup, +1.9 MB decompressed) and restores it. Backward compat: protobuf defaults absent fields and the validator doesn't version-gate, so older pre-novel backups still restore.
- **Deferred:** turning the "couldn't reinstall (repo missing)" restore-log entries into a guided post-restore onboarding step (see the round-2 "Restore-path onboarding" item).
