# FlareSolverr support

Optional Cloudflare bypass for sites that the in-app WebView can't reliably solve. WebView remains the primary solver — most challenges go through it without ever touching FlareSolverr. FlareSolverr is the escape hatch.

## Why this exists

Cloudflare ships several tiers of bot protection. The lighter JS challenges (e.g. `natomanga.com`) can be solved by the in-app WebView in a few seconds, after which the cleared cookies validate for OkHttp's subsequent direct requests. Some sites — `nelomanga.com` is the canonical example — sit on Cloudflare's stricter bot-management tier where:

- WebView often can't solve the challenge within a reasonable timeout.
- Even if a clearance cookie is obtained out-of-band, replaying it through OkHttp fails because Cloudflare ties `cf_clearance` to TLS / HTTP-2 / `__cf_bm` session fingerprints that OkHttp can't reproduce.

[FlareSolverr](https://github.com/FlareSolverr/FlareSolverr) runs a real Chromium instance on a server you control, solves the challenge there, and serves the response page directly. The fork integrates it as a fallback so users on hard-to-bypass sources have a path forward.

## How the integration behaves

1. Every Cloudflare 403 first goes to the in-app WebView. If WebView solves the challenge, OkHttp retries the original request normally and you're done — FlareSolverr is never invoked.
2. If WebView fails *and* a FlareSolverr URL is configured, the request falls through to FlareSolverr.
3. FlareSolverr returns the **page response itself** (proxy mode), not just cookies. The response was generated by FlareSolverr's own browser session that already cleared the challenge, so we serve that response directly back to the source.
4. After the first FlareSolverr success on a host, subsequent requests within the same app session **skip the 30-second WebView pre-attempt** and go straight to FlareSolverr.
5. A **single FlareSolverr browser session is shared across all calls**. The session-resident browser keeps `cf_clearance` and `__cf_bm` in memory, so most follow-up requests return in 1–3 seconds with no JS challenge re-solving.
6. If the FlareSolverr server is restarted (or its session is GC'd) and our cached session ID becomes invalid, the app transparently creates a new session on the next request — no user-visible error.

We deliberately do **not** save FlareSolverr's cookies to OkHttp and replay them — that path is unreliable for the reasons above.

## Setup

*Settings → Advanced → Network → **FlareSolverr URL**.*

1. **Run FlareSolverr** somewhere reachable from your device on your LAN. The two most common shapes:

   ```bash
   # Docker (recommended)
   docker run -d \
     --name=flaresolverr \
     -p 8191:8191 \
     -e LOG_LEVEL=info \
     --restart unless-stopped \
     ghcr.io/flaresolverr/flaresolverr:latest
   ```

   ```
   # Windows native — download the .exe from upstream releases:
   # https://github.com/FlareSolverr/FlareSolverr/releases
   ```

   Both expose port `8191` by default. See the [upstream FlareSolverr docs](https://github.com/FlareSolverr/FlareSolverr) for advanced configuration (proxies, captcha solvers, etc.).

2. **Verify it's reachable.** From a browser on the same network as your Android device, open `http://<flaresolverr-host>:8191`. You should see the FlareSolverr welcome JSON.

3. **Configure in the app.** Settings → Advanced → Network → FlareSolverr URL. Enter `http://<flaresolverr-host>:8191`. The "Disable FlareSolverr" entry directly below clears the URL when you want to turn it off.

That's it — there's no separate enable toggle. A non-empty URL turns FlareSolverr on as a fallback for any source that hits a Cloudflare 403 the WebView can't solve.

### Reaching FlareSolverr from outside your home network

The simplest way to keep your FlareSolverr endpoint reachable when the device leaves your LAN — without exposing it to the public internet — is a mesh VPN like [Tailscale](https://tailscale.com/) (or alternatives such as [Headscale](https://github.com/juanfont/headscale), [ZeroTier](https://www.zerotier.com/), [NetBird](https://netbird.io/)).

1. Install the mesh VPN client on both the FlareSolverr host and your Android device, signed in to the same network.
2. Note the FlareSolverr host's VPN address — for Tailscale this is either a `100.x.y.z` address or a tailnet hostname like `unraid.tail-scale.ts.net`.
3. In *Settings → Advanced → Network → FlareSolverr URL,* enter `http://<vpn-address>:8191` instead of the LAN IP.

This avoids opening port `8191` on your router and avoids the complexity of running a public reverse proxy with TLS and auth in front of FlareSolverr. Connections only succeed when the device is signed in to the same mesh, so the endpoint isn't reachable from the open internet.

## Performance expectations

| Situation | Typical wait |
|---|---|
| Source the WebView can solve | ~5–8 s for the challenge, then cached cookies make follow-ups instant. FlareSolverr never runs. |
| First request to a hard host on a cold app | ~30 s WebView pre-attempt + ~12 s FlareSolverr solve = ~42 s. |
| Tab switch / pagination on a hard host (same app session) | ~1–3 s. FlareSolverr's session-resident browser keeps `cf_clearance` and skips the JS challenge. |
| FlareSolverr server restart | Next request transparently re-creates a session: ~12 s solve, then back to ~1–3 s for follow-ups. No user-visible error. |
| Cold app restart | The "this host needed FlareSolverr last time" memory is in-RAM only, so the very first request after a cold start pays the WebView pre-attempt again (~42 s). Subsequent requests in the new session are fast. |

## Troubleshooting

**App shows `IOException: FlareSolverr returned HTTP 5xx` or a connection error.** FlareSolverr is unreachable or crashed. Confirm the server is running, the URL is correct, and your device can reach it on the LAN. The app falls back to the same "failed to bypass Cloudflare" error WebView would have surfaced.

**App shows `FlareSolverr error: Captcha detected.`** FlareSolverr ran into a CAPTCHA it can't solve. This is rare on manga sources but can happen if Cloudflare escalates challenge tiers. There's no automated workaround — you can try opening the source in WebView via the source's "Open in WebView" action and solving the CAPTCHA manually there, then come back to the app.

**Every request triggers a new FlareSolverr solve, follow-ups never get fast.** The session feature isn't being reused — most likely the app is being killed between requests, or the FlareSolverr server is destroying sessions on each request (some custom configurations do this). Confirm your FlareSolverr is the standard build (`ghcr.io/flaresolverr/flaresolverr:latest` or the official Windows release), not a fork that auto-destroys sessions.
