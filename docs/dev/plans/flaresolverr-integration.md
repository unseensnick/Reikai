# FlareSolverr Cloudflare bypass integration

The developer-facing record for the optional Cloudflare bypass proxy. The user-facing setup, engine choice (Solverr / Byparr / FlareSolverr), and troubleshooting live in [flaresolverr.md](../../flaresolverr.md); this doc covers only how the integration is built and the choices behind it.

## Goal

Give sources behind Cloudflare's stricter bot-management tiers a working path when the in-app WebView challenge solver can't clear them, without changing the normal request path for everything else.

## Why

Cloudflare ships several tiers of protection. The lighter JS challenges are solved by the in-app WebView, after which the cleared cookies validate OkHttp's subsequent direct requests. The strict tiers are the problem: the WebView often can't solve them in a reasonable timeout, and even a clearance cookie obtained out of band fails when replayed through OkHttp, because Cloudflare ties `cf_clearance` to TLS / HTTP-2 / `__cf_bm` session fingerprints OkHttp can't reproduce. A bypass proxy runs a real browser on a server the user controls, solves the challenge there, and serves the response page back. It shipped alongside the manga merge engine in one Yōkai-era port pass, though it is unrelated to merging.

## Approach

WebView stays the primary solver; the proxy is a fallback. Every Cloudflare 403 first goes to the WebView; only if that fails and the proxy is enabled does the request fall through to it. After the first proxy success on a host, later requests in the same app session skip the WebView pre-attempt and go straight to the proxy.

The proxy returns the **page response itself** (proxy mode), not just cookies: the response was generated by the proxy's own already-cleared browser, so the app serves that back to the source (unwrapping the proxy's JSON viewer back to raw JSON for API sources). Cookies are deliberately not saved to OkHttp and replayed, since that path is unreliable for the fingerprint reasons above. A single proxy browser session is shared across calls and transparently re-created if the server drops it; a sessionless engine (Byparr) is detected from the server banner and driven without session reuse.

All proxy mechanics live in the net-new `FlareSolverrClient` (in `core/common`): the `/v1` request/response DTOs, the proxy HTTP client, the warm shared session, per-host request dedup, cookie install, and per-host User-Agent pinning. Mihon's `CloudflareInterceptor` keeps only the challenge detection and the decide-to-delegate step, inside a small `// RK` island, so the patch on the upstream file stays minimal and greppable.

## Key files

- `core/common/.../network/interceptor/FlareSolverrClient.kt`: all proxy internals (net-new).
- `CloudflareInterceptor`: challenge detection + delegate decision, a `// RK` island on Mihon's file.
- `NetworkHelper`, `NetworkPreferences`, `UserAgentInterceptor`, `AndroidCookieJar`: smaller `// RK` islands (client wiring, the URL/enable prefs, UA pinning, the shared cookie jar).

## Status

Shipped (Roadmap P4, on-device verified). Light-novel sources ride the same network client, so a Cloudflare-protected LN site takes the identical WebView-then-proxy path.

## Decisions & tradeoffs

- **Proxy mode (serve the page), not cookie-replay.** Replaying the proxy's `cf_clearance` through OkHttp fails on the strict tiers, so the app serves the proxy's own already-cleared page back to the source instead.
- **Internals extracted to `FlareSolverrClient`.** Mihon's `CloudflareInterceptor` keeps only detection and the delegate decision inside a `// RK` island; every proxy mechanic lives in the net-new class, keeping the upstream patch minimal and the surface greppable.
- **WebView stays primary.** The proxy is only a fallback, so sources the WebView can clear never pay the proxy round-trip, and a per-host "needed the proxy" memory (in-RAM only) skips the WebView pre-attempt on repeat hosts within a session.
