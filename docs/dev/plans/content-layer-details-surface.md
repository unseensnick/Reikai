# Content layer: the details surface

The first surface cut over in the [content layer architecture](content-layer-architecture.md). Read that first: it holds the goal, the ownership rule, and every locked decision this plan executes against. This doc is the per-surface implementation record for details.

## Goal

One shared, Reikai-owned behavior plus screen drives both manga and light-novel details, over the already-neutral Entry* UI. When this ships, a details change is written once and reaches both content types, and `NovelDetailsScreenModel` is gone (dissolved into shared behavior plus a thin novel adapter).

## Why

Details is the keystone: it is the cleanest adapter and the largest single piece of duplication. The two details ScreenModels are roughly 70 to 75 percent congruent, and their multi-source merge methods are line-for-line twins, yet the shipped Entry* work shared only the UI leaves and left the behavior forked. Proving the adapter seam here is what makes the later surfaces (library, browse, migration, reader) tractable. If the interface is wrong, details is where it surfaces cheaply.

Starting position is better than the architecture doc assumed. The shared UI is already fully neutral and value-typed (`EntryDetailsUiState`, `EntryHeaderUi`, `EntryEditInfoUi`, `TrackerAutofill`, `entryInfoItems`, the two `EntryDetails*Scaffold`s), with per-type divergence expressed only as nullable callbacks and slots. The novel screen already drives that UI directly; the manga screen still routes its body through Mihon's `eu.kanade.presentation.manga.MangaScreen` composable. So the remaining fork is the two ScreenModels plus the manga body UI not yet being on the Entry* scaffold.

## Approach

The invariant that governs every step: `MangaScreenModel` stays live and Mihon-synced. It is never made to implement the shared interface, and Mihon's spine (read, download, filter, sort) is never reimplemented in the shared layer. The `MangaEntryAdapter` is a separate object that reads the live model and forwards to its methods, so a renamed upstream field breaks the adapter at compile time rather than in a pixel hunt. Every per-type reconciliation is resolved inside the manga adapter (the manga model is untouched) or by writing the novel side directly against the neutral shapes.

The full migration runs 0 to 5. Steps 0 to 3 ship and are verified first; the riskier manga body-UI migration (4 to 5) is an isolated second on-device checkpoint.

**Step 0, foundation (pure addition, compile-only).** Shipped as the `EntryId` identity plus the cover-cache centralization only. The neutral behavior contract (`EntryDetailsBehavior`, `EntryDetailsScreenState`, `EntryChapterListUiState`, the capability slots) was moved to Step 2, where it is designed against both fully-mapped ScreenModels at once so the Step 3 manga adapter slots in without interface churn. Land `EntryId` (a sealed `EntryId.Manga(rawId)` or `EntryId.Novel(rawId)`, each exposing `rawId` and `contentType`). Centralize the negative-id disguise: the cover caches (the custom-cover file in `CoverCache` and `MangaCover.vibrantCoverColorMap`) are Long-keyed shared infrastructure, and the `-novelId` negation is a legitimate collision-avoidance projection that stays. So this is not a re-key or a key-space change; one `EntryId.coverCacheKey()` projection (manga positive, novel negated) replaces the scattered inline negation at every cover-cache site (the details color cache, `Novel.hasCustomCover`, the cover screen model, the cover fetcher, novel migration). Behavior-preserving: the keys are unchanged, so no cache reset and no schema or backup change.

**Step 1, extract the twinned Reikai behavior (real dedup, zero upstream fork).** Collapse the code that is already Reikai-own and near-identical on both sides before touching anything structural. The first slice, the merge and multi-source actions (`reorderSources`, `resetSourceOrder`, `splitSources`, `removeSourcesFromLibrary`, `removeAllSourcesFromLibrary`), moves into a shared `EntryMergeActionHost` over a small `MergeManager` interface both merge managers implement, carrying the drift-prone snackbar-with-undo logic in one place. The two per-type differences are injected: novels propagate tracker links before a split (`onBeforeSplit`) and write favorite-only so the merge-undo keeps the original `dateAdded` (`setFavorite`). `selectSource` and `showManageSourcesDialog` deliberately stay per-type: their bodies genuinely diverge (novel pagination and selection reset; different chapter-fetch and ranking-call shapes). The merge actions key on `Long` entry ids on both sides, so no `EntryId` parameterization is needed here. Still to extract in later slices: hidden-chapters (the drift-prone filter logic as a generic `applyHidden`) and the tracker-autofill mapping (the shared `EnhancedTracker` exclusion plus the identical metadata fetch). The edit-info to `CustomInfo` conversion is deferred to Step 2: manga and novels emit different output types (`CustomMangaInfo` vs `CustomNovelInfo`) from different signatures, so sharing it now would be premature abstraction; it collapses cleanly once the novel model dissolves and can take `EntryEditInfoUi` directly. This forks nothing upstream because it is Reikai code on both sides today, and it de-risks the rest by shrinking both models first.

**Step 2, the neutral contract plus the novel adapter, dissolving `NovelDetailsScreenModel`.** First land the contract as pure additions (compile-only), designed against both models: `EntryDetailsBehavior` (the shared action surface), `EntryDetailsScreenState` (Loading, Failed, Loaded), and a new `EntryChapterListUiState`. Loaded composes the existing `EntryDetailsUiState`, the chapter-list state, neutral toolbar flags, and the typed capability slots. The neutral chapter item carries `read`, `bookmark`, a `downloadStateOf(id)` provider, and a pre-formatted `readProgress: String?` (novels format their scroll percent, manga their page position, so the numeric progress semantics never need unifying at this surface), plus a shared `isRecognizedNumber` extension (both sides are already `chapterNumber >= 0`, only the name differed).

Then add a thin `NovelEntryAdapter` implementing the contract. `NovelDetailsScreenModel` stays a `StateScreenModel<NovelDetailsState>` with its state machine untouched (low blast radius); the adapter is a separate object that holds it, maps its `Loaded` to the neutral `EntryDetailsScreenState`, and forwards the neutral action calls to the model's existing methods. This is symmetric with the Step 3 manga adapter, which must be a separate object anyway because Mihon's model cannot carry a Reikai interface. The mapping is what resolves the String source id, packs the neutral three-TriState filter call back to the novel's single `Long` setter, backs the neutral per-chapter download state with the disk-cache downloaded-id set, presents `displayNovel` as the neutral non-null `displayEntry`, and pre-formats each chapter's `readProgress` string so the neutral layer never sees scroll percent versus page index. Only genuinely shared actions live on `EntryDetailsBehavior`; the novel-only ones (the page selector, plugin reload) and per-type navigation and intents (opening the reader, share, WebView) stay on the concrete adapter or the thin screen, so the shared spine never rots into no-op methods. The adapter fills the novel page-selector capability slot: not a downstream toggle, its page index is a live input to the chapter combine, so the slot exposes that flow seam (the page keys plus the current index), with the `selectPage` call staying on the concrete adapter, not carried in state (a callback in immutable state would churn recomposition).

Dialogs stay per-type in this step: the behavior and state become neutral, but `NovelScreen` keeps rendering its own dialogs and the shared dialog dispatcher is deferred to Step 5. Until Step 4 the manga side still renders its dialogs through Mihon's `MangaScreen` composable, so a shared dispatcher now would have no manga side to share with.

This step also folds in the edit-info to `CustomInfo` conversion deferred from Step 1. `CustomMangaInfo` and `CustomNovelInfo` are field-identical except the id, and the manga save already takes the neutral `EntryEditInfoUi`. So normalize the novel save from its seven positional params to take `EntryEditInfoUi` plus a new `EntryEditInfoUi.toCustomNovelInfo(novel)` mapper mirroring the manga one, and switch it to a non-cancellable scope so an in-flight edit is not lost if the screen closes mid-write.

Scope boundary (found by reading the novel screen wiring): the `NovelScreen` rewire to consume the neutral state is **deferred to Step 5**, not done here. The screen already drives the shared-UI seams (scaffold, `entryInfoItems`, `EntryToolbar`, selection bar, FAB), but its chapter rows and dialogs stay novel-typed and cannot go neutral until the shared chapter rows (Step 4) and the shared dialog dispatcher (Step 5) exist. Rewiring the working seams now would only churn code Step 5 re-touches, for no runtime gain before a shared `EntryDetailsContent` consumes the adapter. So Step 2 delivers the contract, the compile-proven `NovelEntryAdapter`, the parity level-ups, and the edit-info normalization; the adapter is first consumed at Step 5.

**Step 3, manga adapter over the live model.** `MangaEntryAdapter` holds a reference to the resolved `MangaScreenModel`, maps `State.Success` to the neutral state, and forwards actions to the model's existing methods. It is where the manga-side reconciliations land, all without touching the model: the per-item chapter download state becomes the neutral provider; the anchor collapse (`mergeDisplayManga` or else `manga`, `mergeDisplaySource` or else `source`) moves out of the UI into the adapter; the neutral filter call fans back out to the three TriState setters; the recognized-number predicate stays behind the interface. It fills the manga capability slots (page previews, related carousel, gallery and adult metadata, E-Hentai account, scanlator filter), exposing the entangled ones as the seams they hook (scanlator applied inside the chapter query, EH inside the favorite toggle).

**Step 4, migrate the manga body UI onto the Entry* scaffold (the gated, riskier half).** The manga screen stops rendering through Mihon's `eu.kanade.presentation.manga.MangaScreen` composable and renders through `EntryDetailsScaffold` plus `entryInfoItems` plus manga chapter rows, driven by `MangaEntryAdapter`. Every section the Mihon composable renders maps to either `entryInfoItems` (shared) or a manga capability slot: the gallery-info card and namespaced tag chips land in the existing `aboveDescription` and `searchMetadataChips` parameters; previews, related, and source chips become slot content. The related-carousel slot also gains a placement mode ([unseensnick/Reikai#51](https://github.com/unseensnick/Reikai/issues/51)): a new Settings, Library, Recommendations option renders the recommendations as a three-dot overflow-menu action that opens the recommendations screen, instead of the inline body section, for users who want the feature without the space cost. It is manga-only, matching the gated novel carousel, and distinct from the existing full enable/disable toggle. The cover-seed theme wrap is preserved. Once nothing renders it, Mihon's manga details presentation composable is deleted (not kept inert) and recorded in the off-path manifest, with `refs/mihon` holding the diff base. This is the highest regression risk in the surface, so it is verified section by section against a build of the current release before the file is removed.

**Step 5, unify the screen shell.** `EntryDetailsContent(behavior)` becomes the single screen body both screens call, hosting the neutral dialog dispatcher (with the full-cover Screen-extension host pattern both sides already use). This is where the `NovelScreen` rewire deferred from Step 2 lands: `NovelScreen` stops reading `NovelDetailsState` directly and delegates to the shared Content, driven by the `NovelEntryAdapter` built in Step 2 (its first real consumer). Two thin Voyager `Screen` classes are kept: their open-keys genuinely differ (`MangaScreen(mangaId: Long)` versus `NovelScreen(sourceId: String, novelUrl: String)`, because a browsed novel has no row id yet), and each resolves its model plus adapter and delegates to the shared Content. One Screen forced through a lambda opener is rejected: Voyager constructor args must be serializable or they crash on state-save.

## Key files

- Target seam home: `reikai/presentation/details/` already holds the neutral UI (`EntryDetailsScaffold`, `EntryDetailsTwoPaneScaffold`, `EntryDetailsColumn` with `EntryDetailsUiState` and `entryInfoItems`, `EntryInfoBox`, `EntryActionRow`, `EntryToolbar`, `EntryEditInfoDialog`). The `EntryId`, `EntryDetailsBehavior`, the two adapters, and `EntryDetailsContent` land here.
- Details twins: `eu/kanade/tachiyomi/ui/manga/MangaScreenModel.kt` (stays live, read by the adapter) and `reikai/presentation/novel/details/NovelDetailsScreenModel.kt` (dissolves).
- Screens: `eu/kanade/tachiyomi/ui/manga/MangaScreen.kt` and `reikai/presentation/novel/details/NovelScreen.kt` (both become thin, delegating to `EntryDetailsContent`).
- Delete-and-manifest target (Step 4): the `eu.kanade.presentation.manga.MangaScreen` composable, once the manga body renders through the Entry* scaffold.

## Decisions & tradeoffs

- **Full migration, 0 to 5, not behavior-only.** Behavior-only would dedup the ScreenModels but leave the manga body UI in Mihon's composable, so a body-UI change would still not reach both types. Going all the way is what delivers the anti-divergence goal for the body. The risk is contained by shipping 0 to 3 first and isolating the manga UI migration as a separate verified checkpoint.
- **Merged-display convention: anchor.** The shared Loaded state carries an always-populated display entry plus a resolved source name, the novel's existing convention. The neutral header already flattens the source to a plain name string, so the shared layer never needs the manga's `Source` object. This costs zero change to the live manga state: the manga adapter applies the same or-else-primary collapse the manga UI runs today when it maps to the neutral state. Chosen over the manga's nullable pair (null means primary) because a single non-null field needs no null branch in shared code.
- **The manga adapter reads the live model; it does not dissolve it.** Dissolving the manga model would fork Mihon's spine and turn every upstream change into a permanent hand-port, the trap the architecture doc rules out. Only the novel model dissolves, because its engine is already fully Reikai-owned.
- **Two Screen classes, one Content.** The open-keys differ by content type and must stay serializable for Voyager, so the screens stay separate and thin while the body is written once.
- **Novel recommendation carousel stays gated.** The related-carousel capability slot is manga-only for now; the novel adapter leaves it empty. Revisited when a novel recommendation source exists.
- **Parity gaps to level up in Step 2 (owner-approved).** Novels gain: the manga multi-select delete-chapters confirm dialog (novels currently bulk-delete with no confirm; per-chapter and swipe delete are direct on both sides already, so only the bulk path is the gap); lifecycle-aware state collection (novels use plain `collectAsState`, manga use `collectAsStateWithLifecycle`); a non-cancellable edit-info save (the novel save runs in a cancellable scope today, so a mid-write screen close can drop the edit, unlike manga); and the per-duplicate "Migrate" action on the duplicate dialog. This is built as true parity, not a shortcut: on-device it was confirmed the novel's only migrate path is the full source-picker screen (a different, batch workflow), whereas manga's duplicate card taps straight into a one-tap in-place confirm. So the novel gets a matching one-tap `MigrateNovelDialog` (mirroring Mihon's `MigrateMangaDialog`) reusing the existing `MigrateNovelUseCase` and `NovelMigrationFlag`, triggered by a card tap in `DuplicateNovelDialog` exactly like manga (tap migrates, long-press opens). It covers **all four** places the duplicate dialog fires (details, browse, global search, and the History add-to-library, matching where manga wires it), through one shared `NovelMigrateHost` controller so the wiring lives in a single component instead of being repeated per surface. Because browse and global search present a fresh, unsynced target, `MigrateNovelUseCase` now fetches the target's chapters from its source first (mirroring manga's `updateMangaFromRemote`, best-effort and skipped when the source is unavailable), so read progress carries over from any add-path, not just details. The force-add-despite-duplicate action is not a gap: manga has it via a duplicate-check-off re-toggle and the novel's `addFavoriteAnyway` is the same net behavior, so it becomes one neutral confirm-add-anyway callback. Genuinely one-sided and gated: page previews, gallery and adult metadata, and the E-Hentai account (adult-source concepts); scanlator filter (until novels carry a translator-group column).
- **The interface is designed against both models up front, novel implemented first.** Both ScreenModels are fully mapped, so `EntryDetailsBehavior` is shaped to fit manga and novels at once even though only the novel adapter is built in Step 2. This spends a little more design effort now to keep the Step 3 manga forwarding-adapter from forcing interface churn and a novel-adapter rework.
- **Dialog rendering stays per-type until Step 5.** The neutral `EntryDetailsScreenState` deliberately does not carry the dialog in Step 2. The screen reads the dialog from the concrete model (the novel keeps its own `NovelDetailsDialog`) and renders it, while the shared behaviour still triggers the shows. The shared `EntryDetailsDialog` union and its dispatcher land in Step 5 with the unified screen shell, because the manga side keeps rendering dialogs through Mihon's `MangaScreen` composable until Step 4.
- **The neutral state is lean: filter/sort/display detail stays per-type, and there is no manga capability payload yet.** `Loaded` carries only `hasActiveFilter` for the toolbar tint; the concrete filter, sort and display values feed the per-type settings sheet, which stays per-type until Step 5, so no neutral sort-mode enum is guessed now. `EntryCapabilities` declares only the novel page-selector slot; manga's slots (page previews, related carousel, gallery metadata, E-Hentai account, scanlator filter) are additive nullable slots added with the manga adapter in Step 3, which is not a breaking change to the shared spine.
- **Dissolving the novel model is reshaping, not deleting.** Its roughly 1300 lines are the novel engine driver and mostly remain; the model stays a `StateScreenModel<NovelDetailsState>` and a separate `NovelEntryAdapter` maps its state and forwards its actions. The dedup win in this surface is the neutral contract plus the one genuine merge (edit-info); the write-once-reaches-both payoff lands at Step 5 when a single shared Content renders both types, not from collapsing the two behavior bodies into one (that would be the rejected reimplement-the-spine trap on the manga side).
- **Deferred out of this surface.** The library negative-id disguise (only the two details caches are re-keyed here), the upstream-sync keep-inert to delete-and-manifest rewrite plus its sync-time diff script (created when Step 4's deletion lands), and the release-cut chores.
