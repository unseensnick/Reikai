# Content layer: the details surface

The first surface cut over in the [content layer architecture](content-layer-architecture.md). Read that first: it holds the goal, the ownership rule, and every locked decision this plan executes against. This doc is the per-surface implementation record for details.

## Goal

One shared, Reikai-owned behavior plus screen drives both manga and light-novel details, over the already-neutral Entry* UI. When this ships, a details change is written once and reaches both content types, and `NovelDetailsScreenModel` is gone (dissolved into shared behavior plus a thin novel adapter).

## Why

Details is the keystone: it is the cleanest adapter and the largest single piece of duplication. The two details ScreenModels are roughly 70 to 75 percent congruent, and their multi-source merge methods are line-for-line twins, yet the shipped Entry* work shared only the UI leaves and left the behavior forked. Proving the adapter seam here is what makes the later surfaces (library, browse, migration, reader) tractable. If the interface is wrong, details is where it surfaces cheaply.

Starting position is better than the architecture doc assumed. The shared UI is already fully neutral and value-typed (`EntryDetailsUiState`, `EntryHeaderUi`, `EntryEditInfoUi`, `TrackerAutofill`, `entryInfoItems`, the two `EntryDetails*Scaffold`s), with per-type divergence expressed only as nullable callbacks and slots. The novel screen already drives that UI directly; the manga screen still routes its body through Mihon's `eu.kanade.presentation.manga.MangaScreen` composable. So the remaining fork is the two ScreenModels plus the manga body UI not yet being on the Entry* scaffold.

## Approach

The invariant that governs every step: `MangaScreenModel` stays live and Mihon-synced. It is never made to implement the shared interface, and Mihon's spine (read, download, filter, sort) is never reimplemented in the shared layer. The `MangaEntryAdapter` is a separate object that reads the live model and forwards to its methods, so a renamed upstream field breaks the adapter at compile time rather than in a pixel hunt. Every per-type reconciliation is resolved inside the manga adapter (the manga model is untouched) or by writing the novel side directly against the neutral shapes.

The full migration runs 0 to 5. Steps 0 to 3 ship and are verified first; the riskier manga body-UI migration (4 to 5) is an isolated second on-device checkpoint.

**Step 0, foundation (pure addition, compile-only).** Land `EntryId` (a sealed `EntryId.Manga(rawId)` or `EntryId.Novel(rawId)`, each exposing `rawId` and `contentType`) and the `EntryDetailsBehavior` interface plus its `EntryDetailsScreenState` (Loading, Failed, Loaded). The Loaded state composes the existing `EntryDetailsUiState`, a new `EntryChapterListUiState` (the chapter region plus a per-chapter download-state provider), neutral toolbar flags, typed capability slots, and a neutral `EntryDetailsDialog` sealed type. `EntryDetailsDialog` carries the union of both sides' dialog cases; manga-only cases (set-fetch-interval, EH remove-favorite) and novel-only cases (page selector, chapter settings) coexist, and each adapter only ever emits cases its type supports, so the shared `when` stays exhaustive. Centralize the negative-id disguise: the cover caches (the custom-cover file in `CoverCache` and `MangaCover.vibrantCoverColorMap`) are Long-keyed shared infrastructure, and the `-novelId` negation is a legitimate collision-avoidance projection that stays. So this is not a re-key or a key-space change; one `EntryId.coverCacheKey()` projection (manga positive, novel negated) replaces the scattered inline negation at every cover-cache site (the details color cache, `Novel.hasCustomCover`, the cover screen model, the cover fetcher, novel migration). Behavior-preserving: the keys are unchanged, so no cache reset and no schema or backup change.

**Step 1, extract the twinned Reikai behavior (real dedup, zero upstream fork).** Collapse the code that is already Reikai-own and near-identical on both sides before touching anything structural. The first slice, the merge and multi-source actions (`reorderSources`, `resetSourceOrder`, `splitSources`, `removeSourcesFromLibrary`, `removeAllSourcesFromLibrary`), moves into a shared `EntryMergeActionHost` over a small `MergeManager` interface both merge managers implement, carrying the drift-prone snackbar-with-undo logic in one place. The two per-type differences are injected: novels propagate tracker links before a split (`onBeforeSplit`) and write favorite-only so the merge-undo keeps the original `dateAdded` (`setFavorite`). `selectSource` and `showManageSourcesDialog` deliberately stay per-type: their bodies genuinely diverge (novel pagination and selection reset; different chapter-fetch and ranking-call shapes). The merge actions key on `Long` entry ids on both sides, so no `EntryId` parameterization is needed here. Still to extract in later slices: hidden-chapters (the drift-prone filter logic as a generic `applyHidden`) and the tracker-autofill mapping (the shared `EnhancedTracker` exclusion plus the identical metadata fetch). The edit-info to `CustomInfo` conversion is deferred to Step 2: manga and novels emit different output types (`CustomMangaInfo` vs `CustomNovelInfo`) from different signatures, so sharing it now would be premature abstraction; it collapses cleanly once the novel model dissolves and can take `EntryEditInfoUi` directly. This forks nothing upstream because it is Reikai code on both sides today, and it de-risks the rest by shrinking both models first.

**Step 2, novel adapter and dissolve `NovelDetailsScreenModel`.** The novel screen is already on the Entry* scaffold, so this is the lower-risk adapter and it is fully reversible. The novel model's behavior folds into the shared spine plus a thin `NovelEntryAdapter` that fills the novel capability slots (the paged-source page selector, the plugin-installer ensure-loaded lifecycle) and maps Loaded to the neutral state. Writing the novel directly against the neutral shapes is what resolves the String source id, the packed single-call filter setter, the title-first cross-source predicate, and the `NovelChapter` and text-progress differences. The page selector is not a downstream toggle: its page index is a combine input into the chapter flow, so the capability slot exposes that seam, not just a picker. This step also folds in the edit-info to `CustomInfo` conversion deferred from Step 1: normalizing the novel save to take `EntryEditInfoUi` (as the manga side already does) collapses the two conversions into one shared mapping over a per-type target factory.

**Step 3, manga adapter over the live model.** `MangaEntryAdapter` holds a reference to the resolved `MangaScreenModel`, maps `State.Success` to the neutral state, and forwards actions to the model's existing methods. It is where the manga-side reconciliations land, all without touching the model: the per-item chapter download state becomes the neutral provider; the anchor collapse (`mergeDisplayManga` or else `manga`, `mergeDisplaySource` or else `source`) moves out of the UI into the adapter; the neutral filter call fans back out to the three TriState setters; the recognized-number predicate stays behind the interface. It fills the manga capability slots (page previews, related carousel, gallery and adult metadata, E-Hentai account, scanlator filter), exposing the entangled ones as the seams they hook (scanlator applied inside the chapter query, EH inside the favorite toggle).

**Step 4, migrate the manga body UI onto the Entry* scaffold (the gated, riskier half).** The manga screen stops rendering through Mihon's `eu.kanade.presentation.manga.MangaScreen` composable and renders through `EntryDetailsScaffold` plus `entryInfoItems` plus manga chapter rows, driven by `MangaEntryAdapter`. Every section the Mihon composable renders maps to either `entryInfoItems` (shared) or a manga capability slot: the gallery-info card and namespaced tag chips land in the existing `aboveDescription` and `searchMetadataChips` parameters; previews, related, and source chips become slot content. The cover-seed theme wrap is preserved. Once nothing renders it, Mihon's manga details presentation composable is deleted (not kept inert) and recorded in the off-path manifest, with `refs/mihon` holding the diff base. This is the highest regression risk in the surface, so it is verified section by section against a build of the current release before the file is removed.

**Step 5, unify the screen shell.** `EntryDetailsContent(behavior)` becomes the single screen body both screens call, hosting the neutral dialog dispatcher (with the full-cover Screen-extension host pattern both sides already use). Two thin Voyager `Screen` classes are kept: their open-keys genuinely differ (`MangaScreen(mangaId: Long)` versus `NovelScreen(sourceId: String, novelUrl: String)`, because a browsed novel has no row id yet), and each resolves its model plus adapter and delegates to the shared Content. One Screen forced through a lambda opener is rejected: Voyager constructor args must be serializable or they crash on state-save.

## Key files

- Target seam home: `reikai/presentation/details/` already holds the neutral UI (`EntryDetailsScaffold`, `EntryDetailsTwoPaneScaffold`, `EntryDetailsColumn` with `EntryDetailsUiState` and `entryInfoItems`, `EntryInfoBox`, `EntryActionRow`, `EntryToolbar`, `EntryEditInfoDialog`). The `EntryId`, `EntryDetailsBehavior`, the two adapters, and `EntryDetailsContent` land here.
- Details twins: `eu/kanade/tachiyomi/ui/manga/MangaScreenModel.kt` (stays live, read by the adapter) and `reikai/presentation/novel/details/NovelDetailsScreenModel.kt` (dissolves).
- Screens: `eu/kanade/tachiyomi/ui/manga/MangaScreen.kt` and `reikai/presentation/novel/details/NovelScreen.kt` (both become thin, delegating to `EntryDetailsContent`).
- Delete-and-manifest target (Step 4): the `eu.kanade.presentation.manga.MangaScreen` composable, once the manga body renders through the Entry* scaffold.

Step 0 (the neutral `EntryId` plus centralizing the cover-cache disguise, `24a1a4308`) shipped. Step 1's first slice, the shared merge / multi-source actions over `EntryMergeActionHost` and the `MergeManager` interface, shipped and was verified on-device; its remaining extractions (hidden-chapters, tracker-autofill, edit-info) are pending. The first full-surface on-device checkpoint is after Step 3, the second after Step 5.

## Decisions & tradeoffs

- **Full migration, 0 to 5, not behavior-only.** Behavior-only would dedup the ScreenModels but leave the manga body UI in Mihon's composable, so a body-UI change would still not reach both types. Going all the way is what delivers the anti-divergence goal for the body. The risk is contained by shipping 0 to 3 first and isolating the manga UI migration as a separate verified checkpoint.
- **Merged-display convention: anchor.** The shared Loaded state carries an always-populated display entry plus a resolved source name, the novel's existing convention. The neutral header already flattens the source to a plain name string, so the shared layer never needs the manga's `Source` object. This costs zero change to the live manga state: the manga adapter applies the same or-else-primary collapse the manga UI runs today when it maps to the neutral state. Chosen over the manga's nullable pair (null means primary) because a single non-null field needs no null branch in shared code.
- **The manga adapter reads the live model; it does not dissolve it.** Dissolving the manga model would fork Mihon's spine and turn every upstream change into a permanent hand-port, the trap the architecture doc rules out. Only the novel model dissolves, because its engine is already fully Reikai-owned.
- **Two Screen classes, one Content.** The open-keys differ by content type and must stay serializable for Voyager, so the screens stay separate and thin while the body is written once.
- **Novel recommendation carousel stays gated.** The related-carousel capability slot is manga-only for now; the novel adapter leaves it empty. Revisited when a novel recommendation source exists.
- **Parity gaps to level up, resolved per behavior as it is touched.** Novels gain the manga delete-chapters confirm dialog (novels currently delete with no confirm) and adopt lifecycle-aware state collection (novels use plain `collectAsState`). The force-add-despite-duplicate action is not a gap: manga already has it via a duplicate-check-off re-toggle, and the novel's named method is the same behavior, so it becomes one neutral confirm-add-anyway callback. Genuinely one-sided and gated: page previews, gallery and adult metadata, and the E-Hentai account (adult-source concepts); scanlator filter (until novels carry a translator-group column).
- **Deferred out of this surface.** The library negative-id disguise (only the two details caches are re-keyed here), the upstream-sync keep-inert to delete-and-manifest rewrite plus its sync-time diff script (created when Step 4's deletion lands), and the release-cut chores.
