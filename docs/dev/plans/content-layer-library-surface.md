# Content layer: library surface

## Goal

Collapse the manga and novel libraries onto one Reikai-owned content layer, so the library behaves identically for both content types and a library change is written once. This is the library surface of the [content-layer program](content-layer-architecture.md) (after the details surface), built to the same shape: a neutral state, a behavior seam, adapters over the live engines, an `EntryId` identity, and typed capability slots.

## Why

Unlike details, the library UI is already largely shared. Novels render through the same `LibraryTab` and the same grid, hopper, category picker, and settings by disguising each novel as a negative-id `LibraryItem`, and the grouping / sort / filter kernels (`LibraryDynamicGrouping`, `reikaiSortCategories`, `EntryDisplayPage`, the hopper/picker/fast-scroll grid) are already content-neutral. Two forks remain, and both are exactly the anti-divergence hazard the program exists to remove:

- **The ScreenModel pipeline.** `NovelLibraryScreenModel` reimplements the manga library's `buildState` pipeline (filter, merge-collapse, tracker union, tracker-status filter, sort, dynamic grouping, custom-info overlay), retyped onto `Novel`. Its own code notes it mirrors the manga library's tracker filter. Every library behavior change is two edits, and the two sides drift.
- **The identity disguise.** Novels ride as negative-id fake `LibraryItem`s so they can reuse the manga-typed shared UI and pipeline. The neutral `EntryId` (already used by the details and cover-cache layers) is unused in the library, so the library still leans on the sign-of-the-id trick the architecture doc wants retired.

## Approach

Three phases, lowest-risk first, each independently shippable and on-device verified (debug and minified), matching the program's per-surface discipline.

**Phase 1: the type-agnostic shared transforms.** A deep side-by-side of the two `buildState` pipelines showed the transform stages are logically congruent, but the filter, merge-collapse, and item-build stages are coupled to the entry type (manga transforms `LibraryItem`; the novel model transforms `LibraryNovel`, collapses into `CollapsedNovel`, then builds `LibraryItem`, and the two collapse at different pipeline positions, manga before filter, novel after). So the type-coupled bulk moves to Phase 2, where the neutral `EntryId`-keyed row lands and both sides share one type. Phase 1 extracts only the genuinely type-agnostic pieces, each safe and independent: the **lewd filter level-up** (a genre-based novel `isLewd`, see the decision below), the **shared sort comparator** (the 11 keys are identical across `LibrarySort` and `NovelLibrarySort`; unify to one comparator builder or a decode-to-shared seam), and the **shared tracker union + mean + tri-state filter** helper (both models compute the same merge-group tracker union and include/exclude tri-state). Two open questions to resolve first: whether manga's `getTracksPerManga` is already merge-group-aware, and whether to unify the two sort enums or keep a thin decode seam.

**Phase 2: neutral state, adapters, `EntryId`.** Introduce a neutral `LibraryScreenState` (rows keyed by `EntryId`, capability slots) and a `LibraryBehavior` seam, with a `MangaLibraryAdapter` over the live `LibraryScreenModel` (fenced in `// RK` islands, so a renamed upstream field breaks the adapter at compile time) and a `NovelLibraryAdapter` over the novel engine. `LibraryTab` consumes the neutral state and behavior instead of today's `active*` fall-through (an informal adapter already). Identity moves from the signed `Long` / id-sign branches to `EntryId`, retiring the disguise. This is code-only: no schema migration and no backup change, because novel rows already carry real positive ids and `EntryId.coverCacheKey()` already centralizes the one cover-cache negation. A one-time reset of any runtime cache or UI state keyed by the old disguised id (scroll, selection) is expected and is never a backup concern.

**Phase 3: parity and reconciliations.** Unify the collapsed-set representation (the novel side's single `Set<String>` vs the manga side's split real / dynamic sets), reconcile `NovelLibrarySort` against `LibrarySort`, and finalize the lewd gating and label.

**What stays forked, deliberately (the seams).** The manga and novel engines never merge (different domain models, `String` vs `Long` sources, plugin host vs Mihon sources). The merge-collapse helpers stay forked (different output types, a prior locked decision in [merge-component-consolidation.md](merge-component-consolidation.md)). Download-count derivation stays per-type until the separate download-subsystem unification (Road B). These are injected seams, exactly like the details adapters' `resolveSources` / `setFavorite` lambdas.

### Engine-step ground truth (2026-07-23)

Verified against current code before starting the engine. These are the facts the engine design turns on:

- **The two pipelines convert to the shared row at opposite ends.** Manga builds `LibraryItem` inside `getFavoritesFlow`, then collapses, filters and sorts over it. Novels collapse into `CollapsedNovel`, filter over that, sort over `LibraryNovel`, and call `toLibraryItem` last. The shared kernels are genuinely shared as functions, but each side binds a different `T`, and the novel bindings are pre-conversion domain types. A mixed list has to compare a manga row against a novel row at the sort stage, so this is the constraint the engine design turns on, not categories or source ids.
- **The novel row is already nearly complete as the shared row.** Its synthetic `Manga` carries status, genre, categories, bookmark and read counts and the dates, and the novel status codes line up 1:1 with `SManga`'s. The gap is `source`, left at the factory default, so shared code reading `libraryManga.manga.source` on a novel gets a meaningless value.
- **Novel selection re-runs the whole pipeline for nothing.** `selection` is an input to the main combine but is only written into the state at the end, never read by filter, sort or grouping, so every selection tap re-runs collapse, filter, sort, grouping and item build. Manga keeps selection out of its flow entirely. Novel search also lacks manga's `distinctUntilChanged` plus debounce. A memoized language-code cache already works around the symptom.
- **No shared combine above both providers.** `getFavoritesFlow` combines the manga download-cache tick with two unconditional full-table batch reads (search tags and titles) plus a merged-unread query on every emission. The reads are database batches, not a network scan, and are not gated on metadata sources; today the two download caches are separate classes, so novels cannot fire them. Merge the two providers' upstream flows and they would.
- **Dynamic grouping was only half All-ready.** Bucket ids are index-assigned in one call and collapse keys on name, both fine for a mixed list, but the item id was a raw `Long` shared by the metadata maps, so a mixed call would have de-duplicated a manga and a novel sharing a row id and cross-read their source, language and status. Fixed by making the kernel generic in the id (`fc29c35b9`).
- **The category id spaces really are disjoint.** `categories` seeds a real row 0 with a delete-guard trigger; `novel_categories` seeds nothing and synthesizes Default at render. Both are plain rowid primary keys allocating independently. `LibraryCategoryRef` (`fc29c35b9`) pairs a category with its content type so the mix-up is unrepresentable.
- **The engine is an evolution of one line.** Both models and both adapters are already constructed and both states collected eagerly; `LibraryTab` just picks one. The engine replaces the pick with something that can also merge. All 22 `isNovels` branches live in that one file.

Each step is independently compilable and verifiable, and carries its own tests rather than depending on an upfront test pass. That shape is forced by what is testable: the novel comparator, both collapse helpers, the sort-override decoder and the grouping kernel are pure and reachable, but the manga comparator and filter predicates are private local functions inside `LibraryScreenModel` and `applySort` reads an Injekt-constructed `trackerManager`, so they cannot be pinned until the step that extracts them. Bulk-action expansion needs the whole ScreenModel and is verified on-device, not by unit test. Library-pipeline tests run under `:app:testDebugUnitTest`, not `:domain:test`.

## Decisions & tradeoffs

- **Match the details architecture** (neutral state + behavior seam + adapters + `EntryId`), not a lighter pipeline-only helper, so the whole content layer is one cohesive shape rather than two. Owner-chosen.
- **Extract shared behavior before dissolving the novel model** (Phase 1 before Phase 2), per the program's rule, so each step is incremental and never gambles the whole surface on one big adapter.
- **Retire the negative-id disguise via `EntryId`** (Phase 2). It works today and is code-only, but leaving it forks identity from the rest of the content layer; unifying it is the point of the full collapse.
- **Level novels up on the lewd filter (genre-based), not gate it.** Novel sources carry no nsfw flag (neither the plugin metadata nor the novel model has one, and LNReader has no adult concept), but adult genre tags (`mature`, `adult`, `smut`, `erotica`) are common across the novel plugin ecosystem, so the existing genre heuristic in `MangaLewd` classifies them. Only the manga source-name branch is inapplicable to novels. In-app copy should note that novel lewd-detection is genre-tag-based, so it is weaker than a hard source flag.
- **`LibraryScreenModel` and `LibraryTab` stay live and `// RK`-patched** (engine / partially-collapsed files), never dissolved or added to the off-path manifest while a live remainder exists, per the [upstream-sync](../upstream-sync.md) policy. **Amended 2026-07-22, scoped to this surface:** the shared layer takes over selection and pipeline orchestration, because the "All" chip's mixed manga plus novel list makes a selection neither ScreenModel can own (a range-select spans both types). `LibraryScreenModel` still stays live and on the render path as the manga provider, keeping its favourites flow and its action verbs (read, download, delete, categories) and its Mihon interactor calls. Full reasoning in [content-layer-architecture.md](content-layer-architecture.md), "Decisions & tradeoffs".
- **The merge-collapse helpers and the download-count derivation stay per-type seams**, not shared code.
- **Unify sort via a decode-to-shared-comparator seam, not a flags migration.** `LibrarySort` and `NovelLibrarySort` store the Downloaded and TrackerMean bits with swapped values in `Category.flags` / `NovelCategory.flags`, so one shared decoder would misread every category saved with those two sorts. Each side keeps its own decoder and feeds one shared comparator, resolving the swap before the shared layer: code-only, no persistence change, no backup break. A migration was rejected: it rewrites user data (and needs the same swap in the restore path) for a cosmetic gain the shared comparator already delivers.
- **Retire the id-sign decisions first, the negative leaf id later.** The `EntryId` identity (shipped) routes the shared cover/badge decisions off the id's sign; the leaf row still carries the negative synthetic id as a mechanical key, retired when the pipeline stages are shared and the Mihon pager cells re-key onto `EntryId`.
- **The shared row is Mihon's `LibraryItem`, and novels level up to it before filter and sort (2026-07-23, owner-chosen).** The engine step needs one row type at the sort stage, because a mixed list compares a manga row against a novel row there. Three shapes were weighed. Merging two independently sorted per-type lists at the end was rejected: it produces a correct mixed list but leaves both pipelines forked, so a new filter axis is still two edits, which is the "seam drawn too high" failure this program exists to remove. A net-new neutral row type was rejected as the same collapse at higher cost, since every grid composable consumes `LibraryItem` and would need re-typing. So novels convert earlier instead and the shared pipeline runs over `LibraryItem`, which already carries the neutral `EntryId` as its identity and which novels already produce. This is not novels-as-manga: that rejected design is about the storage layer, and novels keep their own tables and engine. Corollary, corrected after reading `LibraryItem` properly: the row cannot be made honest about its source, and should not try. `Manga.source` is a `Long` and a novel source is a String slug, so there is no valid value to put there, which is why the row already carries novel source facts out of band (`badges.coverSite`, `badges.sourceIconUrl`). The live hazard is that `LibraryItem.matches` resolves the source itself through the manga `SourceManager`, so a novel row reaching it would match against a stub. Novels avoid that today only by calling the shared query matcher directly. So the shared pipeline takes source-derived facts (the name for search, the id for `src:`) from a per-entry side channel keyed by `EntryId`, merged by the engine from both providers, exactly as the grouping kernel now takes its metadata maps. That keeps one uniform accessor bundle with no content-type branching inside the shared layer, and no faked source id.

### Pipeline-share rulings (2026-07-21)

A deep pipeline comparison found eleven behavioural differences between the two `buildState` implementations, plus a systemic merge-awareness gap. Rulings, to be applied as the stages are shared:

- **Collapse before filter (manga's order) wins**, but only after the representative is fixed (next bullet). Collapsing first avoids double-counting chapters that grouped sources share, and keeps the row stable when filters toggle. Filtering first (the novel order) lets a group's cover and title change as filters change, and surfaces entries whose "unread" chapters the user already read on the primary.
- **The collapsed representative must honour the persisted source ranking.** Today both collapse helpers pick by most-chapters-then-earliest-added and never see `merge_group_manga.source_priority`, so a manual reorder on the details screen is ignored by the library: the row shows one source's cover, title and unread count while the details chapter list is trunked on another. Fix by reusing the ranking chain the aggregation already implements (per-group override, then the global preferred-source list, then chapter count). Note `merge_group.title_override` / `cover_override` exist but are never written, and would be the cleaner long-term home for an explicit pinned title/cover.
- **Bulk selection actions expand to the whole merge group, except downloads.** Mark-read and change-categories act on the anchor only today, on both content types, because the collapse drops members out of the emitted list and every action reads the anchor-only derivation. Both models already carry the expanded derivation; it was written for the delete flow and never propagated. Expansion becomes automatic for those two; delete keeps its explicit checkbox (destructive) but defaults on for merged selections.
- **Downloads must never fan out across a merge group.** Grouped sources carry the same chapters, so downloading every member fetches each chapter once per source and burns storage on near-duplicates. The correct target is the group's deduplicated list, which is what the details screen already downloads: it acts on the visible list (the "All" aggregation, or one source when a chip is selected) and groups by each chapter's own `mangaId`, so every chapter is fetched once from its origin source and the ranking override is respected. The library has no equivalent, since it cannot build the aggregation on the selection path, so library downloads stay on the collapsed primary. That approximation improves on its own once the collapse honours the persisted ranking (the primary becomes the user's chosen trunk); making it fully aggregation-accurate belongs with the group-aware next-unread work in [merged-read-state.md](merged-read-state.md), which builds that machinery.
- **Sort unifies via a decode-to-shared-comparator seam**, per the existing decision, reconciling three real differences: the alphabetical tiebreak stays ascending under a descending sort (manga's form, matching upstream), zero-unread entries sort last regardless of direction (manga's form), and Random becomes the per-id seeded comparator (the novel form, the only one expressible as a comparator rather than a short-circuit around one).
- **Tracker mean drops non-positive scores** (the novel form: an unrated tracker is unscored, not zero) and **dedupes by tracker id** (the novel form). Manga's guard tests the raw track list rather than the mapped score list, so an entry whose only tracks belong to a logged-out tracker averages an empty list to `NaN`, which is non-null and therefore escapes the `-1.0` fallback and sorts above every real score. This is upstream's bug verbatim; fix locally.
- **Tracker-status grouping becomes merge-aware on manga** (it reads the primary's tracks only, unlike its own filter and sort), and both sides pass `trackingStatusOrder`, which neither does today, so the intent ordering the grouping kernel implements never fires.
- **Novel library search levels up** to the manga matcher's capabilities (`id:`, `src:`, description, source name, comma-separated negatable terms). The metadata-source tag grammar simply never matches a novel, which is acceptable rather than a faked feature.
- **The library unread badge keeps the representative's own count for this phase.** The merged (deduplicated) count is a scoped follow-up, not part of the pipeline share, because it needs a new table and a `.sqm` migration, a different risk class from the Kotlin-only stage sharing.

### Merged unread count (follow-up, designed not built)

A chapter counts as **read when any member source's copy of it is read**, applied consistently to both the library badge and the details "All" chapter list, so the two never disagree.

Cache the dedup **mapping**, never the count. Read status is not an input to either aggregation, so reading can never change which chapters are duplicates, whereas the count changes on every chapter read and would be stale almost immediately. A canonical-chapter set per merge group (new table beside `merge_group_manga`, with a novel twin) is rebuilt only on chapter-list changes, membership changes, source-order override changes, preferred-source changes and excluded-scanlator changes; the count is then a live SQL join over that set and the `read` column, the same cost class as the aggregates `libraryView` already computes.

A pure-SQL count with no cache was considered and rejected: it works for manga, whose dedup key is the chapter number narrowed to `Float`, but not for novels, whose key is a Kotlin-normalized title string. Taking it would have re-split the two content types, the exact failure this initiative exists to prevent. Recomputing the aggregation per library emission was also rejected: it loads full chapter rows per member, so a library with a few dozen merged groups adds on the order of a hundred queries per emission, and emissions fire on every preference change and DB invalidation.
