# Merge system rebuild

## Goal

One merge system serving both manga and light novels, built around a real, persisted group identity instead of a per-call derivation from preferences plus same-title favorites. Membership becomes a stored fact keyed by entity id, so the library, the details screen, and the reader all read the same group, grouping is created deliberately (never silently), and a whole class of bugs disappears at the root: silent membership corruption, lost unmerges on restore, disagreeing group definitions, browse pre-merging things you never added, and a full-library scan on every details and reader open.

## Why

The current design (`reikai/domain/MergeGroupAlgebra`, `reikai/domain/manga/MangaMergeManager`, `reikai/domain/novel/NovelMergeManager`) never persists a group. It recomputes membership every time from four SharedPreferences sets (`manga_manual_merges` / `manga_manual_unmerges` and the novel twins in `ReikaiLibraryPreferences`) plus a live same-title scan of the whole favorites list. It works, and it bought real things: no schema, instant reversibility, auto-merge-by-title with zero setup. A scout in this cycle verified the costs against current code, and they are no longer acceptable. Every problem below was confirmed by reading current code, not from memory.

- **A live id-reuse corruption path.** `mangas.sq` and `novels.sq` both declare `_id INTEGER NOT NULL PRIMARY KEY` with no `AUTOINCREMENT`, so SQLite reuses `max(rowid)+1` after the top row is physically deleted, and nothing prunes merge prefs on removal (`MangaMergeManager` states sibling removal is intentionally not there; the library `deleteManga` path only sets `favorite = 0`). The trigger is narrower than "any unfavorite": unfavorite keeps the row, so the id is freed only by a physical delete, which happens on the user-run Clear-database (`deleteNonLibraryManga`, deletes `favorite = 0` rows) or on migration. But the hazard is real: a stale pref pair can capture an unrelated new manga that inherits the freed id. `MangaMergeManager.applyMergePrefHealing` only mitigates when both entries are tracked on the same service with disagreeing remote ids; an untracked phantom survives.
- **Restore silently destroys deliberate unmerges.** Merge groups round-trip correctly (serialized as stable `{url, source}` refs and id-remapped, `BackupMangaMerge` / `BackupNovelMerge`), but unmerges are gated by `takeIf { size >= 2 }` and, on the manga side, resolve only against favorites. A dropped unmerge is invisible, and since `autoMergeSameTitle` defaults on, a pair the user deliberately split re-merges itself on restore. Restore also unions with local prefs and never prunes. (One asymmetry the earlier notes missed: `NovelBackupCreator` resolves non-favorite members via a `getById` fallback, so the novel side is not favorites-gated the way manga is.)
- **Two non-equivalent definitions of the group.** `MergeGroupAlgebra.computeGroupIds` unions manual merges with same-title siblings; `MangaMergeCollapse` buckets by merge key `?:` title, so an item that has a manual-merge key is bucketed by that key and its title is never consulted. Worked example: manual-merged A + B and same-title C are one group on the details screen, but a two-cover group plus a standalone C in the library. The library and the details screen disagree about who is in a group.
- **Browse pre-merges copies you never added.** Because `computeGroupIds` always includes the opened id and unions it with every same-title favorite, opening a not-yet-added copy from Browse renders it merged into your library group purely by title (reproduced on a MangaFire copy of a series whose other sources were in the library). The intended behavior is that Browse shows only the source's own details; grouping should be an explicit act, not a title coincidence.
- **A hot-path cost.** Resolution calls `getFavorites.await()` (the whole library) plus, on manga, N parallel tracker fetches, uncached, on every details open and every reader open. (The pref write is conditional on healing actually dropping something, not on every open, so that part of the earlier framing was overstated; the scan and tracker fetches are the real cost.)
- **No group identity.** No id, no row, no owner. Scope cannot be expressed (opening "the group" versus "this source" is indistinguishable to the reader, which forced an Intent-flag patch that was scouted and abandoned in favour of this rebuild). A group cannot own state: no per-group preferred source, title, cover, or cross-source resume. Preferences and the DB cannot commit atomically, so membership changes hand-roll optimistic rollback.
- **`source_order` is overloaded** as a global reading-order index, a per-source column repurposed. The reader sorts on `source_order` alone, so the merge stitcher overwrites it onto one 0..N scale; any list built outside the one blessed function collides the two scales and breaks prev/next (shipped bug, fixed in `3498c37eb`, closes unseensnick/Reikai#49). Manga restamps descending, novel details ascending, so the two sides even disagree on direction.
- **It is roughly 1.2 systems, not 1.** Only `MergeGroupAlgebra` (pure id math) is truly shared. Managers, aggregation, collapse, chips, manage-sources dialogs, backup models, and about eight preference keys are duplicated per content type, against the standing unified-content goal. Novels are the weaker half: the novel reader trusts a caller-passed `orderedChapterIds`, and History and Updates open it with the two-argument reader constructor (in `HistoryTab` / `UpdatesTab`), passing none, so a merged novel opened from either silently degrades to one source.

A real regression net already exists and must be preserved: `MergeGroupAlgebraTest`, `MangaMergeManagerTest`, `NovelMergeManagerTest`, `MergedChapterProviderTest`, `MangaMergeCollapseTest`, `NovelMergeCollapseTest`, plus `MangaMergeBackupRoundTripTest` and `NovelBackupRoundTripTest`.

## Approach

Give the group a persisted identity shared by both content types, and make every screen resolve membership by looking up "which group is this id in?" instead of re-deriving it. Once membership is a stored row, the library scan and the two-definitions split both disappear, FK cascade removes the stale-pref capture, browse stops pre-merging (a non-member resolves to just itself), and scope falls out of identity (opening the group and opening a member become different things, no flag). Auto-merge-by-title stops being a silent live force and becomes an explicit add-time prompt; grouping is always a real row you chose. The group owns a per-group source-ranking override layered over the retained global default, and its display overrides. (A dedicated group-owned ordering table was designed but parked before Phase 3, see Decisions; reading order stays computed in-memory per open, as both readers already do.)

In plain English first, then the mechanism, phase by phase. The spine is 0 -> 1 -> 2; the rest can flex. The phases are the build sequence, not separate releases: the whole rebuild ships together (big-bang), so no reduced version is ever released.

### Design decisions that shape the schema

- **Two per-type member tables, real cascade.** Manga and novels are separate tables with separate id spaces, and a SQLite column can foreign-key to only one parent, so a single member table cannot cascade to both. The group entity is one `merge_group` table with a content-type discriminator; membership lives in two tables (manga members, novel members), each with `ON DELETE CASCADE` to its own parent. A deleted entity's membership vanishes automatically, which is the structural fix for the id-reuse capture. The order table (below, added in Phase 3) cascades the same way, so a delete can never strand an orphan order row.
- **Group-owned ordering table (designed, parked before Phase 3, see Decisions).** Parked because a scout found its driver already fixed (the merge stitcher no longer overwrites `source_order`) and cross-source resume derivable from read-state, so Phase 3 keeps ordering in-memory. Retained here as the revive design. A per-group order table (`group_id`, `chapter_id`, `position`) holds the reading order; the upstream `chapters` / `novel_chapters` tables are left byte-for-byte Mihon, which keeps future reader syncs clean. `source_order` regains its single per-source meaning. Staleness is handled three ways, strongest first: (1) one transactional writer recomputes a group's order in the same DB transaction as any membership change, so membership and order can never disagree (only possible now that this lives in the DB, not prefs); (2) the order is treated as a rebuildable cache, never a source of truth, so a mismatch is a recompute, not corruption; (3) a cheap validity fingerprint (a hash of member ids, their chapter counts, and the effective source ranking) is checked on open and recomputed on mismatch, which is the path-agnostic catch for the one input that changes outside a membership transaction (new chapters arriving from a library update). A dedicated `reading_order` column on the chapter tables is kept as a fallback design if the extra join ever proves too costly; the same staleness recipe applies to it.
- **Full group ownership, delivered by the rebuild.** The `merge_group` row owns a per-group source-ranking override (the global `preferred_manga_sources` / `preferred_novel_sources` list stays as the default; a group uses its own member ordering only when it explicitly overrides, so the common case is unchanged, and this absorbs the standing "Per-group Preferred-sources override" backlog item), an optional title and cover override, and is the natural home for cross-source resume. Storage lands in the phase that first consumes it: identity, membership, display overrides, and the override flag in Phase 0. A persisted ordering table and any resume pointer are parked (see Decisions): the Phase 3 scout found ordering is fine computed in-memory per open and resume is derivable from read-state. This is build sequencing inside one big-bang release, not a scoped-down v1.
- **Explicit membership retires the unmerge concept.** Unmerge pairs only ever existed to override auto-merge-by-title. With auto-merge gone as a live force, "not merged" simply means "not a member." Unmerges are honored once, at migration, so a pair the user deliberately split is not re-materialized, and are vestigial afterward. This deletes the restore-side unmerge-drop bug rather than porting it forward.

### Phases

- **Phase 0: group persistence (foundation, no behavior change).** New migration (`30.sqm`, mirrored into the `.sq` files) for `merge_group` and the two per-type member tables, with the cascades above and the group-owned columns whose storage is known now (title / cover override, the per-group source-ranking override flag, and `source_priority` on each member). The order table and any resume pointer are deferred to Phase 3, where their shape is validated against real queries (migrations are additive, so this costs nothing). Repository plus model in `reikai/domain/merge` + `reikai/data/merge`, registered in DI; no consumers wired yet. Tests: a round-trip that proves membership stores and reads back ordered and that deleting an entity cascades its membership away (the id-reuse fix), plus establishing the missing SQLDelight migration-test pattern.
- **Phase 1: one-time migration (gated on the next `versionCode`, like `SetupCategorySortOverrideMigration`).** Read the four pref sets, reconstruct today's groups (manual entries plus the current same-title auto-groups), honor existing unmerges as exclusions, and materialize everything as rows, so nothing a user has today un-groups. This is the only place the old auto-merge derivation runs.
- **Phase 2: resolution cutover (kills several bugs at once).** Replace `computeRelatedMangaIds` / `computeRelatedNovelIds` derivation with a group-row lookup by id. One definition now serves library collapse, details, and reader, so the two-definitions split disappears; the whole-library favorites scan on every open is gone; FK cascade removes the stale-pref capture; and browse-details of a non-member resolves to just itself (the source's own details), fixing the pre-merge.
- **Phase 3: reader scope.** Give the reader an explicit **scope**, group (the unified cross-source list) or source (one source's own chapters), chosen by the entry point (Option 2, scope-by-context): the details All chip, Library resume, and History open group scope; Updates and a specific source chip open source scope. Both content types obey the one rule, replacing today's accidental split (manga is always group-scoped; novels are source-scoped from Updates / History because those tabs pass no chapter list). The scope signal travels from the entry point (a boolean Intent extra on the manga reader, an explicit parameter replacing the fragile `orderedChapterIds` on the novel reader); the group is still resolved from the entry id, and read-state propagation stays group-wide so marking a chapter read still de-dupes across sources in either scope. Reading order stays computed in-memory per open (both readers already restamp a throwaway copy, never the DB); the persisted order table plus its staleness machinery is **parked** (see Decisions), its original driver (the merge stitcher overwriting `source_order`) already fixed and cross-source resume derivable from read-state. The per-group source-ranking override is deferred to Phase 5 (wiring a consumer before its setter UI exists would ship dead code).
- **Phase 4: backup / restore on the new model.** Round-trip the group entity (members still as `{url, source}` refs), prune on restore instead of blind-union, and drop the unmerge round-trip entirely (the concept is retired). Reuse the existing round-trip tests as the gate.
- **Phase 5: unify and make grouping visible.** Add the explicit add-time prompt (a new action on the shared duplicate dialog: "Add to existing group" vs keep separate, on both manga via `DuplicateMangaDialog` and novels via `DuplicateNovelDialog`). Wire the per-group source-ranking override into `ChapterAggregation.aggregate` (the group's member `source_priority` when `override_source_ranking` is set, else the global list) together with its setter UI, a per-group reorder in the manage-sources dialog, so consumer and producer land together instead of shipping a dead consumer in Phase 3. Collapse the parallel managers, collapse helpers, chips, and manage-sources dialogs onto the shared layer. Retire the old merge / unmerge / auto-merge pref keys (kept read-only so old backups still restore); the global preferred-source keys stay.

Rollout is big-bang: the cutover flips everything to rows, verified on-device, then shipped as a preview release to shake out real-world problems before a stable cut.

## Decisions & tradeoffs

- **Ground-up and both content types from the start**, rather than manga first. Novels are already the weaker half, and doing manga alone would entrench the duplication the unified-content goal exists to remove.
- **Persisted identity, not Komikku's model.** Komikku gives the group identity via a real manga row with a synthetic source plus a `merged` join table, which buys `source == MERGED_SOURCE_ID` as a single dispatch predicate but costs fake rows every query must know about. A group table gives the same identity without them.
- **Two per-type member tables (chosen over a single discriminated table).** A single member table cannot foreign-key-cascade to two parents in SQLite; the per-type split buys a true `ON DELETE CASCADE`, which is the structural fix for the stale-pref capture. Cost: some SQL-layer duplication, inherent to two id spaces.
- **Order computed and cached in a group-owned table: designed, then parked before Phase 3 (2026-07-17).** A Phase 3 scout confirmed the two things that justified the table are already handled. First, the merge stitcher no longer overwrites `source_order`: both readers restamp the merged reading order onto an in-memory `.copy` only (`MergedChapterProvider.restampReadingOrder`, `NovelDetailsScreenModel.restampReadingOrder`), never writing it back, so the DB column already carries its single per-source meaning (the pre-fix overwrite the Why section describes was closed in `3498c37eb`). Second, cross-source resume is derived live from read-state (first unread chapter in aggregated order), not a stored pointer, so no persisted order is needed for it. Phase 3 therefore keeps ordering in-memory per open and ships only the source-ranking override, the novel History/Updates degradation fix, and reader scope. The table (transactional recompute writer plus validity fingerprint, kept above as the revive design) stays parked; revive it only if on-device use over a few days surfaces a concrete need: recompute cost on a hot path, or a persisted order/scope the reader must page through consistently. Tracked in ROADMAP Parked.
- **Per-group source ranking overrides, not replaces, the global list.** The global `preferred_manga_sources` / `preferred_novel_sources` ranking stays and covers the common case; a group falls back to it unless it sets an explicit override (a boolean on `merge_group` plus `source_priority` on its members). This keeps existing behavior on upgrade (every migrated group defaults to the global list, so nothing per-group is computed at migration) and folds the standing ROADMAP "Per-group Preferred-sources override" item into the rebuild.
- **Auto-merge-by-title becomes an explicit prompt.** It stays useful (it still catches same-title dupes at add time) but stops being silent: a reporter on unseensnick/Reikai#49 insisted they were not merged while the code had grouped them by title. After the rebuild, everything grouped is a visible row the user chose, surfaced through the existing duplicate dialog.
- **The unmerge concept is retired**, not ported. Explicit membership makes it unnecessary except as a one-time migration exclusion, which deletes the restore-side unmerge-drop bug at the source.
- **Reader scope by entry point (Option 2, decided 2026-07-17).** Opening a merged-group chapter shows either the unified group list or one source's own list, picked by where you tapped: Details / Library / History open group (continue the series), Updates / explicit source-open open source (stay in that source's feed). Rationale: Updates is inherently a per-source event and is where the group's chapter-number dedup bites hardest (a non-trunk source's new chapter is dropped from the unified list, so group scope would open a chapter that is not in its own list, the case `MergedChapterProvider.withOpenedChapter` had to patch around); History is a reading-timeline resume, so it continues the series. Both content types obey the one rule, ending today's accidental split (manga always group; novels source-scoped from Updates / History). This supersedes the earlier `merge-aware-manga-reader.md` Intent patch rather than reviving it: that extra had to double as group identity, whereas now identity is resolved from the entry id and the signal carries only scope. The mechanism that design got right (scope the reader's `chapterList`, keep `mergedGroup` full so the group-wide read-state pass still de-dupes) carries over.
- **Out of scope (adjacent, not this rebuild):** duplicate DB rows from URL mismatch (a source handing back a different URL for a series already stored makes a second row that no grouping system pulls in automatically). That is a duplicate-detection / URL-canonicalization concern; the add-time title prompt catches it on add, but the rebuild does not solve row-level duplication itself.
