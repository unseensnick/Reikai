# ViewModel migration (Voyager ScreenModel to AndroidX ViewModel)

> **Status: deferred, and this plan is a snapshot.** The migration was scoped and attempted once, then reverted and parked pending an upstream release. Everything version-specific below (file lists, counts, the exact `private` models, the phase split) is pinned to mihon `c3b99aea0` as it looked when this was written; that commit is unreleased and may change before it ships. Treat the [durable approach](#approach-durable) and [durable lessons](#durable-decisions-and-lessons) as authoritative, and re-derive the [snapshot](#snapshot-scope-and-phases-as-of-c3b99aea0) against upstream's actually-released version on [resume](#resume-procedure).

## Goal

Take Mihon's migration off Voyager's `ScreenModel` onto AndroidX `ViewModel` (mihonapp/mihon#3594), across Reikai's own screens as well as the Mihon ones upstream's diff covers, and retire the `voyager-screenModel` dependency. No user-facing change; the payoff is staying on upstream's screen architecture so every later sync applies cleanly.

## Why

It is the one unported upstream change, and it is not a routine sync. Upstream removes the `voyager-screenModel` dependency, adds a `:core:viewmodel` module, and renames every `*ScreenModel` / `rememberScreenModel {}` to `*ViewModel` / `viewModel<T>()`. Reikai has its own screen models that upstream's diff does not cover at all, plus Mihon model files carrying `// RK` patches. Deferring it means every future sync of any screen file fights a base that no longer matches. That standing cost is the reason to do it eventually; the reason not to do it yet is in Status.

## Status

**Deferred.** The migration was attempted once on `feat/0.4.0`: the foundation, the leaf screens, and part of the browse cluster were built and verified on a minified `preview` build on-device. That attempt was then reverted off the branch and preserved in a **local-only backup branch** (`backup/feat-0.4.0-viewmodel-migration`), leaving `feat/0.4.0` at its pre-migration groundwork (the two inert-reference restores and the copy-not-move rule, which stand on their own).

Deferred because upstream's own change is brand new and **unreleased** (no Mihon tag contains it) and still ships a crash-on-open bug in its `private viewModel<T>()` models (see [durable lessons](#durable-decisions-and-lessons)). Finishing now would lock Reikai into a large divergence against code upstream may still revise, and sitting half-migrated makes every unrelated sync awkward.

**Sync consequence:** the synced base stays before the migration commit; the migration is skipped and later upstream commits are ported around it when independent. Recorded in [upstream-sync.md](../upstream-sync.md) under deferred upstream changes.

**Resume trigger:** upstream ships the migration in a release and fixes its own `private` models.

## Why this plan is a snapshot, not a fixed spec

Upstream's migration commit is unreleased. Before it ships, upstream can rename files, add or drop models, fix its `private`-model bug, or restructure the change, and Reikai's own tree also keeps moving underneath it. So the concrete inventory here (which file becomes which, how many models, which four models are `private`, exactly how the phases split) is a **photograph of one moment**, not a contract. It will drift.

What does not drift is the reasoning: why a bare `viewModel<T>()` resolves without a factory, why a `private` one crashes, why per-screen scoping is free under Voyager, why the phases have to be vertical, and which of Reikai's own structures (the `MangaDexFollows` subclass, the drift-heavy library files, the merge code) are traps regardless of upstream's exact diff. Those are properties of AndroidX `ViewModel`, kotlinc, R8, and Reikai's architecture, not of any particular upstream commit.

So the plan is written in two layers. The durable layer is authoritative and should be trusted on resume. The snapshot layer is a strong starting map, but the [resume procedure](#resume-procedure) re-anchors it to upstream's released diff before any of it is followed literally.

## Approach (durable)

**The default factory does the work, and Reikai's Injekt pattern survives untouched.** A model whose constructor arguments are all Injekt defaults is resolved with a bare `viewModel<LibraryViewModel>()`, no factory. A model needing real arguments (a `mangaId`, a `sourceId`) gets a `companion object` holding typed `CreationExtras` keys plus a `viewModelFactory`, and the call site passes values through `extras`. A `Context` parameter comes from `Injekt.get<Application>()` instead of `LocalContext`. The replacement base is one file: `StateViewModel<S>` (`mihon.core.viewmodel`), a shape-for-shape stand-in for `StateScreenModel`. No shim and no typealias: a hard rename.

**Per-screen scoping is free, and needs no wiring.** `viewModel()` inside a Voyager `Screen` resolves to that Screen's own `ViewModelStore`, not the Activity's, because `voyager-navigator`'s Android variant hands every Screen an `AndroidScreenLifecycleOwner` (which implements `ViewModelStoreOwner`, provides `LocalViewModelStoreOwner`, and clears the store on pop). This holds only under a Voyager `Navigator`: content hosted outside one falls through to the Activity store silently. `ReaderActivity` and `WebViewActivity` are outside one and are unaffected, because `ReaderViewModel` is deliberately Activity-scoped via `by viewModels()` and `ReaderSettingsViewModel` stays a manual `remember { }` construction (it takes lambdas, so no factory is possible).

**Phases are vertical, one screen cluster at a time.** Reikai's models are constructed inside Mihon files (a Mihon tab builds a Reikai novel model, a Mihon queue screen builds both queue models), so "all Mihon files, then all Reikai files" cannot compile in between. Each phase takes the Mihon model, the Reikai models it constructs, and every call site together. The two systems coexist throughout (both dependencies present, some models migrated and some not), which is what allows phasing; `voyager-screenModel` drops only at the end.

**Each cluster is one commit, compiled, spotless-clean, and opened on-device.** A green build does not prove a screen opens (the `private`-model crash below compiles fine and dies on tap), so every migrated screen is exercised on a minified `preview` build before its phase is called done. Commits cite the upstream PR and the migration commit SHA.

## Durable decisions and lessons

These are the findings worth carrying forward verbatim; none depend on upstream's exact diff.

- **R8 needs no new keep (proven).** AndroidX's shipped consumer rules keep only `<init>(SavedStateHandle)` and `<init>(Application, SavedStateHandle)`; a `-keep class mihon.**` preserves the class, not its members. A bare `viewModel<X>()` on an all-Injekt-defaults model resolves through the reflective `SavedStateViewModelFactory` to a no-arg constructor that nothing keeps, so the risk was real. It was settled directly on the reverted attempt: the minified `preview` pilots opened cleanly and `dexdump` on the preview APK confirmed the no-arg constructors survive. Reikai's existing `ReaderViewModel` survives only by accident of shape (`@JvmOverloads constructor(savedState: SavedStateHandle, ...)` generates exactly the kept constructor). Re-confirm on the first minified build when resumed, but no keep is expected; if one is ever needed it is one line, `-keepclassmembers class * extends androidx.lifecycle.ViewModel { <init>(); }`.
- **A bare-`viewModel<T>()` model must never be `private`.** The default factory instantiates reflectively from `androidx.lifecycle`, and a private top-level Kotlin class is package-private in bytecode, so `newInstance()` throws `IllegalAccessException` the moment the screen opens. This is **not** an R8 trap: the modifier comes from kotlinc, so it fails on every build type, debug included, and a green build hides it. The fix is one word: widen to `internal` (public in bytecode, still module-private in Kotlin), marked `// RK`. An explicit factory is the other valid fix, and is what upstream already does wherever a model takes arguments, which is why only the no-arg ones break. Upstream currently ships this bug on several of its own no-arg models; taking such a file verbatim imports the crash, so check every model upstream declares `private` and resolves bare. Prefer upstream's own fix once it lands, and drop the `internal` divergence then.
- **`flowWithLifecycle` cannot survive inside a ViewModel.** A ViewModel outlives the composition, so a `Lifecycle` captured from it is the dead Activity's after a rotation. Upstream removed all of its `flowWithLifecycle` calls for this reason. Reikai's own instances are the custom-info overlay and the gallery-metadata flow, both cheap and self-triggered, so dropping the lifecycle parameter is safe; the flows then collect until the Screen is popped rather than pausing on navigate-away. The one flow worth watching is any chapter-list combine that takes `downloadCache.changes` / `downloadManager.queueState` unfiltered, since it re-fires on app-wide download activity; if that measures as real, scope those two inputs to the open entry (as the download flows already are) rather than reintroducing a lifecycle. Rejected alternatives: `ProcessLifecycleOwner` (a permanent divergence in the hardest-to-merge file) and `stateIn(WhileSubscribed)` (rewrites the worst file's state model and diverges from upstream forever).
- **`ioCoroutineScope` is deleted, per upstream's own rule.** It was `Dispatchers.IO`; `viewModelScope` is `Dispatchers.Main.immediate`. Rewrite the sites that launch real work to `viewModelScope.launchIO` and the `stateIn` / `cachedIn` sites to plain `viewModelScope` (their scope only hosts collection; the flows do their own IO). This is compiler-enforced: `ioCoroutineScope` is an extension on `ScreenModel`, so it stops resolving the instant a class extends `StateViewModel`, and no site can be missed silently.
- **A subclass that overrides a parent model needs its own factory.** Kotlin companion objects are not inherited, so a model that extends another and overrides methods (Reikai's `MangaDexFollowsScreenModel` extends `BrowseSourceScreenModel` and overrides its paging + metadata hooks) will, if it leans on the parent's factory, construct the parent type and silently lose every override, with no compile error. Give each such subclass its own key(s) + factory. This is a property of Reikai's class hierarchy, so it applies whatever upstream renames the parent to.
- **Drift, not markers, is the real hazard in the big files.** Reikai's library and details models and tabs are well over upstream's length, so upstream's diff maps to them as a spec to re-lay by hand, not a patch to apply by line. Budget these as re-writes guided by the upstream diff, and drift-check the result against the upstream post-commit blob (only RK-attributable hunks may remain).
- **Merge code is out of scope for the surfacing pass.** The merge-system rebuild ([merge-system-rebuild.md](merge-system-rebuild.md)) replaces or absorbs the merge surfaces and is ground-up across both content types; roughly a third of the details model's Reikai lines are merge handling. Extracting them during this migration is work the rebuild discards plus a conflict handed to it. Likewise leave blocks entangled with Mihon's private state (merged-chapter init, group refresh, related-mangas loader, merge actions) and the deliberate `EntryDetailsUiState` coupling ([unified-content-ui.md](unified-content-ui.md)).
- **Surfacing buried Reikai logic rides this migration, it is not a separate sprint.** The blocks worth lifting out of Mihon files are exactly the ones the migration renames, so doing them in the same pass avoids a rename-plus-extraction commit that destroys rename detection. It is the `unified-content-ui` initiative applied to files that skipped it (shared leaf, per-type mapper, optional slots, net-new in `reikai.*`), which is explicitly exempt from the no-standalone-refactor rule when planned and sequenced.

## Snapshot: scope and phases as of `c3b99aea0`

Everything in this section is the inventory at the time of writing and **must be re-derived against upstream's released version on resume** (see [why](#why-this-plan-is-a-snapshot-not-a-fixed-spec) and the [resume procedure](#resume-procedure)). It is a map, not a checklist.

**Verified counts (re-count on resume):** ~68 files define a model (66 importing a Voyager model base, plus 3 subclassing an in-repo base: `MangaDexFollowsScreenModel`, `GlobalSearchScreenModel`, `MigrateSearchScreenModel`); 79 `rememberScreenModel` call sites across 53 files, all the trailing-lambda form (zero `tag` / `rememberNavigatorScreenModel`); ~33 Reikai-owned models (`reikai/`, `exh/`), of which `EntryTrackInfoDialog` alone holds eight and three take a `sourceId: String` rather than `Long`; 14 Mihon model files carry `// RK` markers.

**Reference points (re-anchor on resume):** the upstream change is `git -C ../refs/mihon show <the released migration commit>`. When this was written, Reikai's synced base was `c3b99aea0^` exactly, so any difference from upstream's parent was Reikai's own drift with no intervening upstream noise; that clean alignment may not survive later syncs, so re-check it. The factory pattern's worked example is upstream's `MangaViewModel` companion object. The hand-merge hot spots (RK islands sitting in upstream's edit region) were `MangaScreenModel.kt`, `BrowseSourceScreenModel.kt`, `CategoryScreenModel.kt`, `SearchScreenModel.kt`. The drift-heavy files were `LibraryScreenModel.kt` and `LibraryTab.kt`.

**Phase map (re-split on resume if upstream's shape changed):**

0. **Foundation and the R8 proof.** Copy upstream's `core/viewmodel` module verbatim, wire it into `settings.gradle.kts` / the app dependency / the catalog + lifecycle bundle, keep `voyager-screenModel` for now, and migrate two pilots (one no-arg from `reikai.**`, one factory-needing from `exh.**`), then build minified and open both on-device to settle R8 before the bulk is in flight.
1. **Leaf screens.** The marker-free files with no Reikai entanglement (webview, deeplink, clear database, create/restore backup, worker info, extension details + filter, upcoming, the migration feature, `LocaleHelper`, the browse sources/migrate screens). Verbatim copies of upstream's post-commit blobs; carries the first `ioCoroutineScope` removals.
2. **Browse cluster.** The browse-source and search models, sources / sources-filter / extensions, the migrate trio, `GlobalSearchScreenModel` (delete-and-add upstream, marker-free in Reikai), plus the Reikai riders: `ReikaiBrowseScreenModel` and the three `reikai*Tab` builders, `exh`'s `MangaDexFollowsScreenModel` (moves with its base; needs its own factory per the subclass lesson), the batch-add / metadata-view models, and the novel browse / globalsearch / migrate / sources models.
3. **Library cluster.** The library, library-settings and category models plus the library tab, and the novel library / category / update-errors models. Risk is drift, not markers (re-lay the two big files); verify both library view modes (tabbed and hopper) and both content types.
4. **Details and reader-settings cluster.** The details model (largest patched file) with its `MANGA_ID_KEY` / `IS_FROM_SOURCE_KEY` factory; the lifecycle change lands here. Also the cover and notes models, the reader-settings model (Reikai's path is `ui/reader/setting/`, singular), the novel details / cover models, the related-mangas model (the only Reikai model holding a `Context`), and the eight models inside `EntryTrackInfoDialog`. Upstream's `TrackInfoDialog` change lands verbatim on the restored inert reference; hand-apply the same change into `EntryTrackInfoDialog` by diffing upstream-before against upstream-after. Do not copy upstream's `settingsviewModel` typo in `ReaderActivity`.
5. **History, updates, downloads, more.** The history, updates, updates-settings, more, and stats models, and Mihon's inert download-queue model (take upstream's rename; stays inert). Reikai: novel history / updates, both queue models, both bulk-favorite models, and `HistoryScreenModelStateProvider` (moves with History; upstream renames it with a find/replace slip worth correcting rather than copying). Landmines: `MoreViewModel` is one of upstream's `private` bare-`viewModel<T>()` models (make it `internal`), and `NovelBulkFavoriteScreenModelTest` constructs its model directly with a `Dispatchers.Main.immediate` scope, so check whether it needs `Dispatchers.setMain`.
6. **Teardown, docs, ledger.** Drop `voyager-screenModel` from the catalog and bundle once no `cafe.adriel.voyager.core.model` import remains, take upstream's `Navigator.kt` (which deletes the `ScreenModel.ioCoroutineScope` extension and the last `ScreenModelStore` use), rewrite the docs the migration invalidates, append the sync ledger row, add one `Other` line in `CHANGELOG.md`, and do a final minified build.
7. **Inert-marking debt.** A few Mihon composables with no call sites but no inert note read as live and should be marked (as `MangaCoverDialog` is), plus a couple of stale docstrings/imports to clean while the files are open.
8. **Surface the buried bodies.** Lift the Reikai logic sitting in Mihon files into its Reikai-owned home per the surfacing lesson above, excluding merge code and the entangled-state blocks.

**Docs this migration invalidates and must rewrite with it:** [CLAUDE.md](../../../CLAUDE.md) screen conventions 2 through 7, [.claude/rules/screen-conventions.md](../../../.claude/rules/screen-conventions.md) (two headings are named after the API, and its reference screen points at a file upstream renames), [.claude/rules/architecture.md](../../../.claude/rules/architecture.md), [development.md](../development.md), the naming exemplar in [.claude/rules/code-quality.md](../../../.claude/rules/code-quality.md), and the semi-normative `.claude/skills/scout/SKILL.md` + `.claude/skills/port-audit/SKILL.md`.

## Resume procedure

When upstream ships the migration in a release and fixes its own `private` models:

1. **Re-anchor to reality.** Pull `refs/mihon`, find the released migration commit, and read its actual diff. Do not assume it matches the snapshot above.
2. **Re-derive the inventory.** Re-run the counts, re-list the model files and call sites, and re-check which models upstream declares `private` and resolves bare (its fix may have removed some, or moved to explicit factories). Confirm Reikai's synced base still lines up cleanly with the commit's parent; if later syncs have intervened, separate upstream's change from Reikai's drift first.
3. **Re-split the phases** against the released shape if it differs, keeping them vertical (one cluster of Mihon model + Reikai riders + call sites per commit).
4. **Execute per the durable approach:** verbatim upstream blobs where marker-free, hand-merge around `// RK` islands where not, drift-check each hand-merge against the upstream post-commit blob, and open every migrated screen on a minified `preview` build before calling a phase done.
5. **Prefer upstream's fixes over Reikai's stopgaps.** Where upstream has since fixed its `private`-model bug, take that rather than carrying the `internal` divergence.
6. **Close it out** with the teardown, docs rewrite, ledger row, and CHANGELOG line (phases 6 to 8 above).
